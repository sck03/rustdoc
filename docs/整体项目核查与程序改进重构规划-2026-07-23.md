# 整体项目核查与程序改进重构规划

> 审查日期：2026-07-23；增量更新：2026-07-24
> 审查范围：产品架构、运行目录与路径、数据库和依赖、前端多平台体验、字体许可、发布验收。
> 本文性质：基于当前源码和既有文档形成的“审查基线 + 实施结果 + 后续路线图”。本轮已完成前端商业化体验收口和自动门禁更新；未完成的实机、容器和生产式压力验证仍不会写成已完成。

## 1. 结论摘要

项目已经从旧式单体桌面程序完成到“领域层 + 应用层 + 基础设施层 + ASP.NET Core API + React/Vite + Tauri 壳 + Rust Sidecar”的主线迁移，主要业务边界、权限模型、运行数据根、SQLite 单机和 PostgreSQL 多人模式均已有实现与专项文档。

当前更准确的判断是：**架构方向可继续投入，开发期重构已进入收口阶段，但还没有达到全平台生产发布状态**。主要原因不是缺少业务模块，而是发布证据仍分层：Windows x64 证据最完整；Linux/macOS、ARM64、容器首次构建和 Tauri 更新链路仍有待验收；依赖许可证和运行时体积还需要集中治理。前端桌面和平板已经具备较成熟的商业后台骨架；本轮已解决 390px 手机全局导航挤占首屏、HS 双重流程负担、仪表盘纵向冗长、单一窗口空白和报表中宽布局问题，手机端现在可按“查询、查看、轻量审核”产品边界继续验收。

本轮重新从当前源码生成并人工复核 80 个视觉场景，而不是沿用历史截图；批准图更新后又以普通测试模式独立复验。自动视觉、缩放和无障碍契约不仅比较像素，还新增手机菜单默认关闭、首屏业务内容、打开/关闭、遮罩和 Escape 焦点恢复断言。自动门禁仍不能替代真实用户、真机触控和跨操作系统字体验收，但已覆盖本轮发现的具体产品意图。

本轮审查没有发现业务数据库“必须”写入系统 C 盘的实现。`.NET API` 直接运行时 `RuntimeAppPathProvider` 的默认数据根是程序根下的 `App_Data`：如果用户只有一个 C 盘，或明确把程序安装在 C 盘，这是合理且可运行的本地回退；如果设备有非系统盘或企业有数据盘策略，则可以通过 `EXPORTDOCMANAGER_DATA_ROOT`、Tauri 运行目录选择或部署 volume 将数据移出 C 盘。该项属于部署策略差异，不应作为程序 P0 缺陷。

### 4.1 目标目录契约

| 内容 | 稳定位置 | 可写位置/规则 | 审查结论 |
| --- | --- | --- | --- |
| 程序、模板、资源、Noto 字体、浏览器、OCR 模型 | 程序根 | 发布包只读 | 已实现 |
| SQLite 数据库 | 运行数据根 `Database/` | `EXPORTDOCMANAGER_DATA_ROOT` 或 Tauri 选择目录；未指定时回退程序根 `App_Data` | 已实现；支持按部署策略迁移 |
| 日志、缓存、WebView、上传、任务产物 | 运行数据根下对应目录 | 成功/失败/取消后清理临时内容 | 已实现，需做断电残留演练 |
| PostgreSQL 数据 | 部署方显式 volume，例如 `runtime/postgres/` | 不进入镜像层 | Compose 已定义，未在本机实测 |

### 4.2 已确认的例外

1. Tauri Windows 当前检测到系统盘后会要求选择非系统盘目录，并拒绝再次选择系统盘；这会影响只有一个 C 盘的设备。API 直接启动使用 `AppContext.BaseDirectory/App_Data`，该默认值本身适合单磁盘或明确接受程序根数据的单机部署。这里应调整的是 Tauri 的选择策略，不是把程序根 `App_Data` 定义为 P0 路径缺陷。
2. 授权防复制锚点使用 Windows 注册表 + DPAPI、macOS Keychain、Linux Secret Service。它不是数据库或普通业务数据，不能用“全部文件都不写系统组件”作为验收条件；应在隐私说明中明确用途和卸载清理方式。
3. WebView2、操作系统文件对话框、Docker Engine 自身镜像缓存和系统安全存储由宿主平台管理，项目只能控制数据目录和调用边界。
4. 测试中的 `D:/...`、`/opt/...` 等路径是平台命令构造样例，不是生产默认路径；应继续确保这类字面量只存在测试代码。

### 4.3 条件化路径增强建议（不改变业务模型）

- 保持“显式参数/环境变量 > 受控配置 > 程序根 `App_Data`”的解析顺序。只有部署方明确要求禁止系统盘时，才启用强制非系统盘策略；普通单机用户不应因为只有一个 C 盘而无法运行。
- Tauri 继续在 Windows 检测系统盘并给出选择提示，但允许用户确认使用程序根数据目录；服务模式和 Compose 通过环境变量/volume 显式指定数据根。
- 运行检查页面只显示相对目录、磁盘卷标和可写性结果；管理员诊断接口才返回完整绝对路径。
- 给 `Database`、`Security`、`Backups`、`Exports` 设定 ACL、容量阈值和清理策略，清理任务只删除本程序登记的文件，不扫描用户下载目录。
- 增加断电、只读程序目录、只读数据目录、路径配置损坏、跨盘移动和符号链接/重解析点测试。

## 5. 数据库与依赖审查

### 5.1 数据库判断

- SQLite 采用标准开源 SQLite，连接启用外键、WAL、`synchronous=NORMAL`、10 秒 busy timeout 和单实例锁；不引入 SQLCipher，静态保密交给 BitLocker、FileVault、LUKS 和加密备份。
- PostgreSQL 使用 `postgres:18-bookworm`，数据挂载到宿主运行根；API 任务、会话、权限和业务数据不另建 Redis、消息队列或旁路 JSON（SQLite 的任务快照属于明确的单机例外）。
- Compose 示例把数据库密码写入运行目录的 `config/appsettings.json`，该文件必须由部署脚本设置最小 ACL、加入备份加密，并在正式部署优先迁移 Docker secrets/受保护环境变量；不得提交 `.env`、配置和数据库到 Git。
- 当前没有数据迁移兼容旧生产库的要求，开发期可以按当前模型重建；但每次结构升级仍需提供幂等启动检查、备份说明和失败回滚策略。

### 5.2 当前主要依赖

| --- | --- | --- | --- |
| UI | React、React Router、TanStack Query、Lucide、Three.js | MIT/ISC 等宽松开源许可 | 保持；只在确有交互收益时引入新库 |

| HTML/PDF | Scriban、HtmlAgilityPack、PDFsharp、PDFtoImage、PdfPig | MIT/BSD/Apache 等 | 功能重叠较大，建立“渲染、解析、像素回归”职责表，避免继续叠加 |

### 5.3 依赖治理计划

1. 建立 `Directory.Packages.props` 或等价中央版本表，统一 .NET 包版本；npm、Cargo 保持 lock 文件并在 CI 使用冻结安装。
2. 每次依赖升级记录上游版本、许可证、原生二进制、体积变化、CVE 状态和回归入口；生成可审查的 SBOM 与第三方 notices。
3. 为 PDF/Excel/浏览器三类重量级依赖设置发布包体积预算，重复 native 库和 Playwright 开发 UI 进入硬门禁。
4. 在生产发布前运行 `dotnet list package --include-transitive`、npm audit（仅审查，不自动升级）和 Cargo audit；漏洞例外要有到期日期和替代方案。
5. 不为了“新”而引入大型 UI 框架。现有 Lucide、TanStack Query、原生 CSS 和 React 已足够；除非需要复杂拖拽、虚拟表格或国际化，否则优先编写小型领域适配器。

### 5.4 Web/容器多人权限与后端稳定性核查

Web/容器版和 SQLite 单机版不能用同一套运行假设验收，当前源码已经形成以下差异化边界：

- 非回环监听必须显式启用 network mode，而且数据库必须是完整配置的 PostgreSQL；启动校验会拒绝把 SQLite 暴露为局域网或容器多人数据库。跨域仍需白名单，正式部署优先同源反向代理。
- Web/容器有效权限由“产品版本上限、部署形态、权限模板、`view / operate / manage` 操作级别、业务数据归属”共同决定。React 负责菜单、路由和控件体验，`ApiWorkspaceAccessMiddleware` 在服务端再次校验并返回 403，不能通过直接调用 API 绕过前端隐藏。
- PostgreSQL 普通账号的发票、付款、CRM、跟进、供应商、商机、装柜方案和单一窗口派生数据按 `OwnerUserId` 过滤；管理员可审查全部。邮件模板区分“本人可编辑、共享可读取”，用户报表模板另支持本人、部门、公司和团队共享范围。
- SQLite 是一台电脑上的单用户/单进程产品模型，一般业务数据不做 PostgreSQL 式账号归属过滤；会话只需随本机进程存在。报表模板等明确具有“我的模板”语义的功能仍保留所有者规则，但这不等于允许多台电脑共享 `.db`。
- SQLite 通过运行数据根锁文件阻止同一数据根重复实例，并启用 WAL、外键、`synchronous=NORMAL`、shared cache、连接池和 10 秒 busy timeout。它适合本机可靠运行，不承担网络并发和共享文件数据库容灾。
- PostgreSQL 会话和后台任务写数据库，会话只保存 token 摘要；连接启用有限瞬时失败重试和有界连接池，初始化使用 advisory lock，避免多个启动流程同时建表或播种。
- 多步骤数据库写入使用执行策略与事务；主要可编辑实体携带 `RowVersion` 或 `VersionNumber`，旧页面保存冲突映射为 HTTP 409。前端查询只对有限瞬时错误重试，普通写操作不自动重放，避免网络抖动造成重复新增或重复业务动作。
- 后台任务具备全局、单用户、浏览器重任务三层运行并发和全局/用户排队上限；任务开跑前重新检查账号启用状态和有效权限，账号排队期间被停用后不会继续执行。取消、超时和异常路径会回收页面、浏览器进程树与受控临时目录。

当前可承诺拓扑仍是“一套 PostgreSQL、一个 API 实例、多个浏览器用户”。多 API 副本需要数据库任务租约、跨实例取消和共享文件存储后才能支持。还必须明确：`PostgreSqlIntegrationTests` 只有设置 `EXPORTDOC_TEST_POSTGRES_CONNECTION_STRING` 才连接真实服务器，未设置时会直接结束，因此本机显示的 .NET `570/570` 不能解释为真实 PostgreSQL 已验收。生产前仍需完成真实数据库初始化、四岗位并发、服务重启、断网/超时、备份恢复、大数据量和长时间运行测试。

### 6.3 本轮已完成：手机导航产品契约

原问题确实来自 `responsiveOverrides.css` 在相同断点重新把 `.nav-list` 强制为横向 `display:flex`，覆盖了主题层的关闭状态，并非截图过期。本轮已经删除冲突规则并把 860px 以下导航改为默认关闭的覆盖式纵向抽屉：

- 初始只显示品牌、菜单按钮和当前页面上下文，主业务内容直接进入首屏。
- 打开后显示遮罩，点击遮罩、按 Escape、完成路由跳转均可关闭；背景滚动被锁定。
- Tab/Shift+Tab 在抽屉内循环，关闭后焦点回到菜单按钮；按钮与抽屉通过 `aria-controls` 建立关系。
- 自动门禁同时断言默认状态、打开/关闭、遮罩、Escape、焦点恢复和首屏内容位置，不再只比较像素。

该项已经从体验 P0 转为真机 P1 验收；尚未覆盖的主要是 320/375px、iOS/Android 返回手势、软键盘、触屏误触和真实读屏。

### 6.4 各主要页面的人工视觉核查

#### 登录页

- 当前结果：账号、密码、高级连接设置和主按钮层级保持不变，页面以静态、低干扰的企业工具结构为基础；中文产品名为第一层级，英文为辅助信息，版本以独立徽标表达，入口图标统一品牌青绿色。2026-07-24 在此基础上增加低频背景漂移、单次入场和短控件反馈，保持克制而非持续高频绘制。
- 删除原因：原 `LoginBackgroundScene.tsx` 是约 459 行纯装饰 Canvas2D 动画，不是 Three.js。它没有登录、连接、权限或品牌识别的业务功能，却持续占用绘制资源、增加低性能设备和“减少动态效果”用户的负担，也让截图回归和维护更不稳定。
- 依赖边界：构建中的 `vendor-three` 约 503.90 kB、gzip 126.94 kB，仍只服务按路由加载的装箱模拟器；删除登录动画不会削弱装箱三维功能。
- 已落实选择：品牌动感由只改变 `transform/opacity` 的 CSS 微动效承担；手机降低节奏并隐藏次要信号线，`prefers-reduced-motion` 压缩为一次近即时执行，不恢复整套 Canvas 场景。

#### 全局导航与信息架构

- 桌面导航的选中态、分组图标和层级线比较统一，但全功能管理员视角有 9 个顶层分组，首次使用者需要较长时间理解系统边界。
- 用户语言已收口：“知识中心库 / HS 编码库”改为“HS 归类与税则 / HS 编码知识”，“局域网 / 容器协同”改为“多人协作”，“本地优先 · 桌面运行”改为“本机运行”。部署术语继续只放管理员设置和运维文档。
- 顶部同时展示连接状态、账号角色、密度和退出入口，信息完整但略偏管理台。普通业务用户可只保留身份、连接异常和高频入口，正常连接状态不必持续抢占视觉注意力。

#### 仪表盘

- 桌面：六项指标、最新订单和待办事项层级清楚，是目前最接近商业后台成品的页面之一。
- 平板：图标轨道释放了内容宽度，三列指标仍能保持可读，整体表现良好。
- 手机：指标已改为双列紧凑摘要，待办优先于低频数据，避免把六张桌面 KPI 卡纵向逐张放大；仍需用真实长业务数据检查卡片高度和触控密度。

#### 发票编辑

- 桌面：未保存状态、分步页签、单据编号、保存按钮和表单分组明确，用户知道当前处于编辑状态；卡片与字段对齐稳定。
- 手机：表头、主操作、未保存徽标和基础信息已能按单列阅读，比很多宽表页面更成熟。
- 风险：商品明细是高列数、高频录入核心区，首屏基线不能证明完整编辑流程适合手机。应继续以桌面为完整编辑端，手机提供查看、少量字段修正和明确的“建议在桌面完成”边界，避免为了形式上的全功能把几十列强塞进手机。

#### HS 编码知识中心

- 桌面：五步流程清楚，候选编码、来源、状态和确认操作均可见，业务可信度较好。
- 当前结果：候选证据与操作区已经收紧，减少大屏无效空白；手机只显示当前流程步骤，不再同时横向铺开五步流程和六个功能入口。
- 产品边界：年度税则导入、批量审核和证据维护仍以桌面端为主；手机适合查询、查看候选和轻量确认。

#### 单一窗口

- 标准/高级模式区分是合理的渐进式设计，能避免普通用户一开始面对全部专业字段。
- 手机纵向布局遗留的桌面 `flex-basis` 已清除，工具栏、草稿状态和回填动作之间不再出现大块空白；完整专业字段录入仍建议桌面端完成。
- “单一窗口”是行业术语可以保留，但子页面名称应尽量使用办理目标，而不是协议、容器或接口实现语言。

#### 报表设计

- 桌面：设计/预览、模板选择、复制、导入导出和画布边界清楚；管理卡已取消不必要的强制等高，中等宽度时模板包占满整行，减少顶部空白和不自然拉伸。
- 手机：当前明确提示“可选择模板、查看预览并进行轻量确认；完整设计与模板包导入导出请使用桌面端”，这个产品边界正确，应保持；不能把桌面设计器等比例缩到手机。
- 构建结果中设计器页面约 202.65 kB、gzip 49.45 kB，jsPDF 和 html2canvas 也较重。现有按页面分块是正确方向，后续应避免让这些依赖进入普通查询页首屏。

#### 弹窗、错误和状态反馈

- 删除发票弹窗是当前较成熟的细节：标题说明后果、显示单号、红色危险动作和取消动作，手机上也没有按钮拥挤。
- 加载、空数据、普通错误、致命错误、离线、本地离线、服务不可达、权限不足、路由重定向和并发冲突均有独立状态基线，说明异常体验不是事后补丁。
- 后续人工验收重点应转向恢复路径：错误后是否保留用户输入、重试是否幂等、离线恢复是否自动刷新、冲突解决是否能让普通用户理解。

### 6.5 字体、颜色、密度与普通用户可读性

- UI 使用系统字体回退栈：Windows 优先 Segoe UI/微软雅黑，macOS 优先 SF Pro/PingFang，Linux 回退 Noto/system-ui。这些字体名称作为系统回退没有再分发许可问题，但各平台字宽、字重和行高不完全一致，因此不能只用 Windows 截图代表全部平台。
- 正式报表随包使用 Noto CJK，许可证为 OFL 1.1，可免费商用和再分发；UI 与报表字体策略应分开记录，不要把系统专有字体复制进安装包。
- 当前 11 个 CSS 文件仍存在大量 `12px/13px` 字号字面量。它们适合高密度专业后台，但对于普通用户、中老年用户和 125%--200% 系统缩放环境，不能仅以“缩放脚本通过”判断舒适度。正文、表单值和关键状态应尽量维持 14--16px；12px 主要用于非关键说明和标签。
- 全局青绿色识别度较强，危险红、警告琥珀和状态绿也基本明确；登录入口图标已从亮蓝统一为品牌青绿色。视觉系统仍应减少无业务含义的渐变和发光，把色彩留给主操作、选中态、状态和风险。
- 按当前源码做出现次数扫描，11 个 CSS 文件约有 1506 个十六进制/RGB/HSL 颜色字面量、100 个阴影和 43 个渐变。数量不直接等于质量问题，但说明 token 覆盖仍不彻底，后续换肤、暗色模式和跨模块一致性会受到重复规则影响。
- 44px 触控目标、手机输入框 16px、焦点和减少动画基础方向正确；还需要真机验证 Windows 高 DPI、macOS Retina、浏览器文字放大、软键盘遮挡和触屏误触。

### 6.6 解耦性与前端可维护性

- Web 与 Tauri 共用 React 页面，桌面能力通过桥接层隔离；业务页面没有直接依赖 WinForms、系统路径或 WebView2 私有接口，这一点适合继续扩展 Web、桌面和容器部署。
- 页面按路由分块有效，业务页大多形成独立 chunk；Three.js 仅在装箱模拟器路由按需加载，jsPDF、html2canvas 和报表设计器也没有全部挤入主包。登录页不再维护独立 Canvas 动画运行时，轻量动效归属现有 CSS 层且没有新增脚本生命周期。
- `foundation.css`、`workspaces.css`、`responsive.css` 目前主要是导入壳，底层仍有 `styles.css`、`theme.css`、`responsiveOverrides.css` 等大文件和重复断点。手机导航的具体级联冲突已修复并有门禁，但跨文件重复定义同一选择器的结构风险仍需继续收敛。
- 后续不建议为了“更现代”整体换成新的大型 UI 框架。现有 React、TanStack Query、Lucide 和原生 CSS 足以完成商业化收口；新库只应解决明确缺口，例如经测量确有需要的虚拟表格、拖拽或国际化，并通过许可证、体积、维护活跃度和跨平台验证。
- 大页面继续按“页面协调器 + 查询/mutation hook + 纯业务模型 + 展示组件”拆分；响应式规则应归属到组件或单一布局层，避免多个全局文件竞争相同选择器。

## 7. 字体和免费商用审查

- `NotoSansCJKsc-Regular.otf`、`NotoSansCJKsc-Bold.otf`、`NotoSerifCJKsc-Regular.otf` 均由 manifest 固定 commit 和 SHA-256，许可证为 SIL Open Font License 1.1，许可证全文和 README 随包保留，适合免费商用和再分发。
- `.otf` 不提交 Git，而在构建阶段下载到程序根；这避免仓库膨胀，但发布必须使用可复现下载、哈希校验和离线缓存，不能依赖用户机器联网。

- 当前字体文件约 56 MB，属于可接受但需预算的发布资产。后续可评估按平台/语言拆包或使用子集化，但不能在没有中文长文本、繁简、标点、货号和报表回归的情况下直接裁剪。
