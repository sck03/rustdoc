# 审计日志维护交互重构与跨平台审查记录

> 更新日期：2026-07-15
>
> 适用范围：React/Vite Web 前端、Tauri 桌面壳、ASP.NET Core API sidecar、SQLite/PostgreSQL 当前运行数据库。

## 1. 本轮结论

审计日志原界面把“按筛选条件删除”和“按保留期清理”压缩在一行，并让两个动作都要求输入 `DELETE`。这会让普通管理员误以为两者只是同一个删除动作的不同按钮，也无法在点击前确认实际影响范围。

本轮完成了以下统一规则：

| 动作 | 实际范围 | 是否受当前筛选影响 | 新确认方式 |
| --- | --- | --- | --- |
| 删除筛选结果 | 仅删除已提交查询命中的记录 | 是 | 显示命中数量与筛选摘要，确认弹窗点击“确认删除” |
| 按保留期清理 | 删除早于保留天数的历史记录 | 否 | 明确显示保留天数和“当前筛选不参与”，确认弹窗点击“确认清理” |

删除筛选结果必须至少存在一个筛选条件，服务端拒绝无条件删除全部日志；历史维护应使用保留期清理。两种操作均永久删除，不提供回收站或程序内恢复。

## 2. 代码重构

- `AuditLogPage.tsx` 只负责查询状态、分页、API mutation 和页面编排；维护动作移至 `AuditLogMaintenancePanel.tsx`。
- `auditLogMaintenanceModel.ts` 集中维护“是否存在有效筛选”和筛选摘要生成，可脱离 React 单独测试。
- 新增通用 `ui/ConfirmationDialog.tsx`，支持 Escape 关闭、焦点回到取消按钮、遮罩点击取消、窄屏按钮纵向排列和危险/警告语义。
- 维护区拆为“导出当前结果”“按保留期清理”和折叠的“高级操作：删除筛选结果”，按钮使用文字与图标并列，不再依赖只有图标的危险操作。
- 保留 React、Lucide 和现有 CSS 体系，没有引入新的 npm、NuGet、Cargo、Python 或云服务依赖。

## 3. API 契约与数据边界

- `ApiAuditLogDeleteRequest` 和 `ApiAuditLogCleanupRequest` 使用 `confirmed: boolean` 表达确认意图，删除魔术字符串 `DELETE` 和前端确认输入框。
- 导出接口按运行形态拆成两个明确契约：`POST /api/audit-logs/save-to-path` 只允许可信 Tauri desktop token，写入用户在桌面保存对话框选定的路径；`POST /api/audit-logs/download` 仅返回 Excel attachment，不接收目标路径、不创建服务器导出副本，适用于浏览器和容器。
- 删除端点验证 `confirmed=true` 且至少有一个筛选条件；清理端点验证 `confirmed=true`、保留天数大于 0。
- OpenAPI Schema、生成的 TypeScript 客户端和集成测试同步更新。项目尚未投产，因此直接移除旧 `/export` 命名和旧 DTO，不增加兼容别名、双写或运行期猜测分支。
- 审计日志仍只读写当前运行数据根数据库的 `AuditLogs` 表；桌面路径保存只写管理员明确提供的 `.xlsx` 路径，浏览器下载在内存生成并以标准 Excel MIME 返回，不读取发票、报关、付款或报销业务表。

## 4. 多平台与响应式审查

- Web 页面继续使用平台无关的 React/CSS；维护区在宽窗口使用双卡片布局，在 `860px` 以下自动单列，确认弹窗和危险操作在窄屏纵向铺满。
- Tauri 桌面保留文件选择、打开导出文件和运行目录路径桥接；浏览器不依赖 Tauri 全局对象即可渲染查询和维护说明。
- 新增代码没有使用 WinForms、WebView2 API、反斜杠拼接、系统用户目录、`AppData`、`ProgramData`、系统临时目录或硬编码盘符。
- 运行目录、数据库、日志、缓存、WebView、备份和业务文件继续遵循“程序根稳定资源 + 运行数据根可写数据 + 用户显式输出目录”。
- 健康检查的绝对路径仅向可信 desktop token 返回；桌面 smoke 已同步携带 token，匿名 Web 健康检查继续脱敏。
