# 新版报表设计器可执行方案

> 生成日期: 2026-07-02
>
> 依据: `设计规划.md`、`程序改进文档.md`、`多平台Tauri重构可执行方案.md`、`运行目录与路径存储审查清单.md`、`程序改进重构进度文档.md`。

## 1. 目标结论

报表模板是本系统的核心资产之一，不应长期停留在“类 Excel 表格设计器 + HTML 源码编辑 + 预览”混合形态。当前设计器继续保留作为稳定入口；新增“新版报表设计器”作为实验入口，先支持发票模板最关键的打印版式能力，跑通后再逐步替换旧设计器。

新版设计器不直接采用 GrapesJS 或 Craft.js 作为最终模型。原因是外贸单证模板更接近固定纸张报表，而不是网页/邮件/低代码页面。新版设计器应采用“结构化报表 schema + React 画布 + HTML/Scriban 输出”的路线。

第一阶段目标:

- 拆出独立前端模块 `features/report-designer`。
- 在报表模板页新增“新版设计器”实验入口。
- 支持发票模板的纸张、文本、字段、明细表、分页符和预览。
- 保存仍输出现有 `.html` 模板，不改变 API、数据库和模板目录。
- 旧设计器保留，遇到复杂模板可继续回退编辑。

## 2. 设计原则

- **模板格式不锁死**: 继续使用当前 `HTML + Scriban + Chromium PDF` 主线，不引入商业私有报表格式。
- **多平台优先**: 设计器完全在 React/Web 层实现，Tauri、浏览器和后续 Docker/Web 部署均可复用。
- **固定纸张优先**: 以 A4/A5、横版/竖版、边距、页眉、主体、页脚和分页为核心，不做通用网页设计器。
- **业务组件优先**: 提供文本、字段、固定票据表格、明细表、条件块、分页符、图片/印章等单证常用组件，限制随意网页布局。
- **预览即事实**: 预览继续调用现有报表渲染 API，确保设计器看到的 HTML 与最终 PDF 来源一致。
- **直接替换旧可视化入口**: 新版设计器承接可视化模板设计，源码模式仍保留给高级维护；旧 Luckysheet iframe、静态资源和离线文件不再作为运行入口。
- **不新增默认落点**: 不新增数据库表、缓存目录或系统 C 盘/AppData/ProgramData 默认路径；模板仍读写程序根 `Templates/`。

## 3. 成熟产品参考与取舍

新版设计器应借鉴成熟产品的“报表模型”和交互组织方式，但不能直接照搬通用网页编辑器或商业报表格式。参考方向如下:

| 方向 | 参考价值 | 本项目取舍 |
| --- | --- | --- |
| JasperReports / FastReport / ActiveReports / Stimulsoft | 参考页眉、明细 band、页脚、分组、分页、小计、跨页表头和固定纸张模型 | 借鉴报表概念和属性面板结构，不直接引入私有模板格式，不把现有 HTML/Scriban 模板迁入商业格式 |
| jsreport Studio | 参考 HTML 模板、Chrome PDF、在线模板管理和预览闭环 | 与本项目当前 `HTML + Scriban + Chromium PDF` 路线相近，可重点参考“模板管理 + 数据样例 + 预览”的工作流 |
| ReportBro | 参考 Web 报表设计器如何处理固定纸张、字段、表格、PDF 和属性面板 | 可参考固定页面和报表元素模型，但本项目仍输出现有 HTML/Scriban，不引入独立运行时 |
| GrapesJS | 参考左侧组件、画布、属性面板、块拖拽和序列化思路 | 只作为交互参考。GrapesJS 默认偏网页/邮件流式布局，不作为本项目报表 schema 或打印模型 |
| Craft.js | 参考 React 组件化编辑器架构、可编辑节点和属性面板绑定方式 | 只作为架构参考。Craft.js 更像搭编辑器的框架，不提供报表分页、表格循环和 PDF 稳定性 |

结论:

- 本项目不做通用网页设计器，不让用户随意搭复杂网页布局。
- 本项目也不直接切换到商业报表引擎私有格式，避免模板资产被锁死。
- 新版设计器以“单证报表常用组件 + 固定纸张 + 结构化 schema + HTML/Scriban 输出”为主线。
- 成熟产品主要用于校准功能边界、信息架构和报表概念，不作为直接依赖。

## 4. 业务化设计器边界

新版设计器的核心定位是“单证报表设计器”，不是“网页装修工具”。它必须限制自由度，把能力集中在外贸单证常用版式上。

必须支持:

- 纸张: A4、A5、横版、竖版、自定义纸张尺寸。
- 页面: 上下左右边距、默认字体、默认字号、打印 CSS。
- 版区: 页眉、主体、页脚；页眉/页脚支持跨页重复，版区可设置最小高度和避免跨页拆分，页脚可设置短页贴近页底。
- 文本: 静态文本、字体、字号、加粗、对齐、边距。
- 字段: 从当前报表类型字段目录插入字段；出口单据使用 `Invoice/Customer/Exporter/item`，付款/报销使用 `Payment.*` 和金额换算字段，两个业务数据域不能混用。
- 多列行: 支持自由设置列数、列宽、列宽条拖动/方向键微调、固定文本/字段、单元格样式和四边边框，用于发票同一行左右不同内容。
- 固定票据表格: 支持自定义列数、列宽、列宽等分、列宽条拖动/方向键微调、行高、增删行、增删单元格、横向/纵向合并、固定文本、字段、勾选组、竖排文字、单元格样式、边框，以及默认单元格样式/表格边框批量套用，用于工厂付款单、费用报销单、内部审批单等固定格票据；不绑定某个固定付款单样式。
- 条件块: 按字段有值、等于或不等于固定值控制一段固定文本或字段内容是否输出。
- 图片/印章: 支持字段图片或受控静态图片地址，使用 `mm` 设置宽高，支持对齐、空源不打印和避免跨页拆分。
- 组件输出: 所有 block 支持“参与输出”开关和设计备注，用于临时停用组件或记录设计意图；停用组件仍保留在画布和报表模型树中，但导出 HTML/Scriban 时不生成实际报表内容。
- 明细表: 结构化列配置，列宽使用 `mm`，字段名绑定到 `Invoice.Items`，支持列宽等分、列宽条拖动/方向键微调、列级边框批量套用和清除列级覆盖，支持分组表头、分组小计和分组另起页，导出 Scriban 循环；该组件只用于出口单据，不插入付款/报销模板。
- 分页: 显式分页符，导出 `page-break-before` / `break-before`。
- 预览: 始终走现有 API 渲染，设计器预览必须接近最终 PDF 来源。

明确限制:

- 不开放任意 HTML/CSS 网页布局作为普通用户入口。
- 不允许第一阶段任意绝对定位和复杂层叠。
- 不把明细表当作自由表格手工画，而是用结构化列生成循环。
- 不在普通设计模式暴露 Scriban 语法编辑；源码模式保留给高级维护。
- 不允许设计器绕过模板保存 API 直接写文件。

这套边界的目的，是让模板设计更接近 Jasper/FastReport 这类报表工具的“组件 + band + 属性”模型，同时保留当前项目的 HTML/Scriban/PDF 技术主线。

## 6. Schema 草案

第一阶段 schema 只覆盖发票模板核心能力，后续再扩展付款/报销和单一窗口相关模板。

第一阶段可把 schema 存在 HTML 注释中，避免新增数据库或独立文件:

```html
<!-- EXPORTDOC_REPORT_DESIGNER_SCHEMA
{ ...json... }
-->
```

保存时生成完整 HTML。旧模板没有 schema 时，新版设计器可只读打开并提示“该模板未包含新版设计器结构，可从空白模板新建或转为基础结构”。

## 7. 画布模型

新版画布采用固定纸张预览，而不是自由网页流布局。

推荐结构:

```text
外层滚动区
  纸张阴影容器
    page-box: A4/A5/mm 尺寸比例
      margin-box
        Header section
        Body section
        Footer section
```

交互规则:

- 左侧侧栏按“组件库 / 报表模型 / 字段目录”组织。组件库只提供受控报表组件，不开放任意网页块；报表模型树按 Header、Body、Footer band 展开 block，可直接选中画布组件或版区；字段目录继续按业务分组插入字段。
- 报表模型树继续借鉴成熟报表工具的对象树，但只显示本项目 schema 中的受控概念。Header、Body、Footer 会标注页眉/主体/页脚 band 角色；模型面板顶部会显示报表类型、纸张、业务数据源、字段绑定数量和停用组件数量；每个组件节点会展开它实际绑定的字段，明细表会继续展开数据源、跨页表头、非循环侧栏、分组表头、分组小计、分组另起页和表尾合计等子节点，帮助用户按报表模型和数据源理解模板，而不是按通用网页 DOM 理解模板。
- 画布和模型树都把 Header、Body、Footer 作为一等报表对象处理。选中版区时，右侧属性面板切换到版区打印属性、最小高度和组件摘要；选中 block 时，右侧属性面板切换到组件属性。
- 组件放置遵循报表模型约束: 文本、字段、多列行、固定票据格、条件块和图片可放入任意版区；明细表作为主体明细 band 只能放入 Body；分页符只能放入 Body，不能放进跨页重复页眉或页脚。拖拽或工具栏插入时会把不合规的新组件归位到可放置版区，已有组件移动到不合规版区时保持原位，保存前校验也会阻断非法结构。
- 设计器主体默认按 A4 竖版纸张高度呈现；左右侧栏与当前纸张高度齐平，内部自行滚动，避免侧栏高度超过主体造成视觉重心漂移。
- 组件按纵向文档流排列，避免第一阶段就做复杂绝对定位。
- 每个 block 可选择、上移、下移、复制、删除。
- 文本和字段支持字体、字号、加粗、对齐、边距。
- 条件块支持选择条件字段、判断方式和输出内容，第一阶段只包裹一段文本或字段，不做复杂嵌套。
- 图片/印章块支持字段图片和受控静态地址，宽高使用 `mm`，普通用户不编辑任意 `img` HTML 或脚本协议。
- 组件属性支持“参与输出”和设计备注。停用输出时组件在画布和模型树中灰显并保留在 schema 中，便于临时比较版式或保留设计说明；导出器不会为该组件生成报表 HTML/Scriban。
- 固定票据表格以受控 table schema 表达，不暴露任意 HTML；用户可通过列宽、列宽等分、列宽条拖动/方向键微调、行高、单元格合并、字段、勾选组、竖排文字、默认样式套用和边框批量套用自行搭出工厂付款单这类票据，而不是只能套用某个固定预设。
- 明细表支持列增删、列宽 mm、列宽条拖动/方向键微调、标题、字段、对齐和边框。
- 分页符显示为明确分隔线，导出为 CSS `page-break-before: always; break-before: page;`。

暂不在第一阶段支持:

- 任意元素绝对拖拽定位。
- 每页小计、承前页/转下页和复杂条件嵌套。
- 富文本混排。
- 多模板批量套版。

## 8. HTML/Scriban 输出

设计器保存时输出当前项目已经支持的 HTML 模板。核心要求是生成结果可被现有 `ReportHtmlService`、Scriban 渲染和 Chromium PDF 复用。

版区打印输出采用 HTML/CSS 表达，不引入 Jasper/FastReport 私有 band 格式。导出器会把 Header/Body/Footer 放入 `edm-report-page-table` 页壳: 页眉跨页重复映射为 `thead { display: table-header-group; }`，页脚跨页重复映射为 `tfoot { display: table-footer-group; }`，版区最小高度映射为受控 `min-height: Nmm`，版区避免拆分映射为 `break-inside: avoid` / `page-break-inside: avoid`。旧 v0/v1 schema 回读时会迁移到 v2，但缺失版区打印属性的旧结构默认保持非重复输出，也不会凭空补版区高度，避免静默改变旧模板打印行为。

字段块输出示例:

```html
<span>{{ Invoice.InvoiceNo }}</span>
```

明细表输出示例:

```html
<table class="edm-detail-table">
  <thead>
    <tr>
      <th style="width: 34mm;">品名</th>
      <th style="width: 20mm;">数量</th>
      <th style="width: 22mm;">金额</th>
    </tr>
  </thead>
  <tbody>
    {{ for item in Invoice.Items }}
    <tr>
      <td>{{ item.ProductNameEN }}</td>
      <td>{{ item.Quantity }}</td>
      <td>{{ item.TotalPrice }}</td>
    </tr>
    {{ end }}
  </tbody>
</table>
```

条件块输出示例:

```html
{{ if Invoice.SpecialTerms }}
<div class="edm-conditional-block">Special Terms: {{ Invoice.SpecialTerms }}</div>
{{ end }}
```

图片/印章输出示例:

```html
{{ if doc_seal_path }}
<div class="edm-image-block edm-image-keep-together" style="text-align: right; margin-top: 4mm;">
  <img src="{{ doc_seal_path }}" alt="Document seal" style="width: 42mm; height: 24mm;">
</div>
{{ end }}
```

分页符输出示例:

```html
<div class="report-page-break-row" style="page-break-before: always; break-before: page;"></div>
```

## 11. 预览闭环

新版设计器预览流程:

```text
schema state
  -> reportDesignerHtmlExporter
  -> HTML + Scriban
  -> 现有 previewReportTemplateContent
  -> iframe srcDoc
```

预览只进入前端内存 iframe，不写数据库、不创建临时文件、不改模板。用户点击保存后才调用现有模板保存 API。

新版设计器右侧预览面板提供两个数据模式，并在样例数据模式下提供样例档案:

- 样例数据: 默认使用当前 schema 导出的内存 HTML 调用 `previewReportTemplateContent`，未保存时也能即时预览；同时提供受控本地样例档案，包括出口单据常规样例、出口长明细分页样例和付款票据样例，用于快速检查字段绑定、明细循环、长文本换行、固定票据格和分页压力。
- 当前单据: 使用报表模板页当前选择的发票/付款报销单和既有 `previewInvoiceReportHtml` / `previewPaymentVoucherHtml` 端点，只在当前 schema 导出 HTML、页面源码和已保存模板内容一致时启用，避免把未保存设计稿误认为真实单据结果。

本地样例档案只在前端内存中对当前导出的 HTML/Scriban 做受控字段替换、简单条件求值和 `Invoice.Items` 循环展开，不新增 API、不写入模板或样例文件，也不作为最终打印事实；最终业务预览仍以既有后端 Scriban 渲染和 Chromium PDF 结果为准。

真实数据预览不要求用户输入发票 ID 或付款 ID。报表模板页按当前报表类型拉取最近业务单据，以下拉形式展示单据编号、客户/收款方、日期和金额；URL 携带的 `invoiceId`、`paymentId` 或 `sourceId` 只作为从业务页面跳转时的当前单据兜底。

## 14. 技术选型建议

新版报表设计器是重新设计的固定纸张报表编辑器，不以旧 Luckysheet 设计器为技术基座，也不把 GrapesJS、Craft.js 这类通用网页/低代码编辑器作为最终模型。旧 Luckysheet 可视化入口和离线资源已清理；新版模块可以引入适合本项目的现代维护库，但必须服务“结构化报表 schema + React 画布 + HTML/Scriban 输出 + Chromium PDF 预览”这条主线。

选型原则:

- 优先选择免费开源、仍活跃维护、TypeScript 友好、React 18/后续 React 版本兼容、无商业锁定的库。
- 优先选择解决单一清晰问题的库，例如拖拽排序、状态历史、画布缩放、表单校验，而不是把报表 schema 交给通用网页编辑器。
- 固定纸张、band/section、字段绑定、明细表循环、分页和 HTML/Scriban 导出仍由本项目掌控。
- 引入依赖前必须评估包体积、维护状态、免费开源许可证、离线打包可用性、Tauri/Web 兼容性和打印/PDF 稳定性。
- 第一阶段已先用 React state/reducer 思路落基础壳，不新增生产依赖；后续交互复杂度上来后再按需引入。

推荐:

- 拖拽排序: 当前已用原生 HTML5 drag/drop 覆盖组件/字段拖入、block 跨版区重排和明细列拖拽排序，列宽可视化调整已用原生 pointer/键盘事件覆盖，不新增生产依赖；后续如需更复杂触控体验、跨容器嵌套拖拽或完整键盘可访问拖拽，再评估 `@dnd-kit/*` 最新稳定版本。
- 状态管理: 第一阶段继续 React reducer/history；当跨面板编辑复杂后评估 Zustand 最新稳定版本，但 schema 变更仍保持可序列化。
- 撤销/重做: 当前先用 history stack；后续可评估 Immer patches 或同类现代 patch 工具，前提是便于 schema migration 和测试。
- 画布缩放/平移: 优先自实现轻量 zoom/pan；若复杂度增加，再评估维护活跃且无布局模型侵入的 React 画布辅助库。
- 表单与校验: 属性面板优先复用现有字段组件；复杂 schema 校验可评估 Zod 最新稳定版本，但校验规则要和后端/导出器保持一致。
- 富文本: 第一阶段不引入；后续若确需富文本块，再评估 Tiptap/ProseMirror 生态的最新稳定版本，并限制输出为可控 HTML 片段。
- 单位: mm/pt 为主，CSS 导出时保留 mm/pt。
- 预览: 继续复用现有 API 和 iframe，正式打印仍以 Chromium PDF 渲染结果为准。

GrapesJS 可参考其 block manager、属性面板和序列化思路；Craft.js 可参考 React 组件编辑架构。但二者不作为新版设计器依赖或报表模型来源，避免被网页编辑器模型绑住。任何旧库、停止维护库或与 Tauri 离线打包冲突的库都不进入新版设计器主线。

## 15. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| 新版设计器无法覆盖旧复杂 HTML | 保留源码模式，旧 HTML 首次结构化保存需确认，不自动转换复杂旧模板 |
| 打印效果与画布不一致 | 预览始终走现有 API + Chromium 渲染，不单独做假预览 |
| schema 演进困难 | schema 带 `version`，已新增 migration/validation helper；后续新增版本时先补 migration 和测试 |
| 用户误覆盖旧模板 | 新版保存前检测是否有 schema；旧模板首次结构化保存需确认 |
| 功能范围膨胀 | 第一阶段只做纸张、文本、字段、轻量条件块、明细表、分页、预览 |
| 目录落点失控 | 不新增文件落点，schema 嵌入 HTML，模板保存仍走既有 API |
