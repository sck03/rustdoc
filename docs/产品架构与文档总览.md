## 1. 产品版本

桌面 SQLite 单机版统一使用同一套代码和数据库模型，通过构建参数生成三种产品版本：

| 版本 | 面向岗位 | 默认工作区 |
| --- | --- | --- |
| 单证员版 `Document` | 单证员 | 发票、付款、报表、单一窗口、单证工具 |
| 业务员版 `Sales` | 外贸业务员 | 销售概览、客户跟进、商机、邮件模板、供应商、汇率和单封邮件工具 |
| 全功能版 `Full` | SOHO、管理人员、小团队 | 单证工作区和销售工作区 |

局域网/Docker Web 版只发布全功能版本，再按账号岗位和权限模板显示并授权模块。`Admin` 使用不可降权的管理员模板，`Sales / User / Finance` 默认绑定各自内置模板，也可由管理员改绑自定义模板；岗位继续负责管理员身份和数据归属，模板负责模块与操作级别，二者不混写。

### 1.1 业务员版导航与权限基线

| 使用身份 | 应显示 | 不应显示 |
| --- | --- | --- |
| `Sales` 产品的本机管理员 | 销售概览、客户跟进、商机跟踪、邮件模板、供应商管理、今日汇率、邮件发送、软件更新、授权注册、系统设置、关于系统 | 单证仪表盘、发票、查询、付款、任务中心、主数据、单一窗口、报表、Excel、OCR、装箱、审计日志、用户与权限 |
| `Full` 产品中的 `Sales` 岗位账号 | 销售工作区、供应商、今日汇率、邮件发送、关于系统 | 全部单证工作区；软件更新、授权注册、系统设置、审计日志、用户与权限等管理员入口 |
| `Full` 产品管理员 | 单证、销售和系统管理全部入口 | 无版本级隐藏项 |

导航隐藏、前端直达路由、设置分类和 API 端点必须使用同一能力口径。销售版不得只隐藏菜单而继续开放单证仪表盘、主数据或自定义候选项 API；用户管理和审计管理只属于全功能版管理员。授权页在试用到期时仍允许系统强制跳转，不把授权恢复路径锁死。

局域网和容器部署固定使用 `Full` 产品能力，桌面 `Document / Sales` 版本不作为服务器岗位。浏览器导航同时受岗位和运行环境约束：Tauri updater 只在可信桌面运行时显示，浏览器管理员不显示“软件更新”。

| 岗位 | 业务导航 | 系统导航 | PostgreSQL 数据范围 |
| --- | --- | --- | --- |
| `Admin` | 单证、销售、供应商、汇率、邮件全部工作区 | 授权注册、审计日志、系统设置、关于系统；不显示桌面软件更新 | 可查看和管理全部业务数据 |
| `Sales` | 销售概览、客户跟进、商机、邮件模板、供应商、汇率、邮件发送 | 仅关于系统 | CRM、跟进、商机、供应商和本人模板按本人归属；共享邮件模板可读取 |
| `User` | 单证仪表盘、发票、查询、付款、任务中心、单一窗口、主数据、报表和单证工具，另含汇率、邮件 | 仅关于系统 | 发票、付款及派生单一窗口数据只访问本人归属 |
| `Finance` | 付款报销、单据查询、报表设计、汇率、邮件、OCR、关于；付款/查询所需候选项、客户、收款方和出口商读取作为隐藏基础能力提供 | 仅关于系统 | 只访问本人归属；隐藏模块的 API 同步返回 403；报表模板按模块管理级别授权 |

项目当前不存在财务概览页面、路由、API 或财务专属统计服务，正式清单不得把“财务概览”写成已实现模块。保留 `finance.dashboard` 作为未来模块键，但在模块真正开发并通过权限/数据范围测试前不进入导航和模板编辑器。Finance 第一阶段现复用付款报销、单据查询、报表设计、汇率、邮件、OCR 和关于；发票编辑、单一窗口、主数据维护、Excel、装箱和全部销售模块已从导航、直达路由和 API 同步隐藏/拒绝。

网络安全边界：匿名和普通账号访问 `/healthz` 只得到版本、状态和数据库模式，不返回服务器绝对路径或依赖位置；完整运行路径只向管理员 Bearer 或可信 Tauri desktop token 返回。通用邮件页在浏览器中只发送正文，任意服务器文件路径附件只允许可信 Tauri 桌面请求，网络版附件必须从受控发票/报表输出入口生成。

### 1.3 Full 多用户版权限模板（第一阶段已实现）

成熟商业 Web/容器版的模块级权限模板已落地，并保持与三种桌面产品版本的边界：

- `Document / Sales` 桌面版继续使用构建版本固定能力，不显示用户和权限模板管理。
- `Full + SQLite` 单机版只向管理员提供同一套高级权限入口，便于小团队预配置账号；普通单机用户无需使用。
- `Full + PostgreSQL` 局域网/容器版由管理员维护内置或自定义模板并分配给账号，是权限模板的主要使用场景。
- 最终有效权限必须取“产品版本上限 ∩ 部署形态 ∩ 权限模板 ∩ 数据归属范围”，导航隐藏不能代替 API 拒绝。

数据契约已经使用独立 `PermissionTemplate / PermissionTemplateModule / User.PermissionTemplateId`，不把大量模块布尔值塞入 `User.Role`，也不为每位用户复制导航 JSON。模块权限提供 `view / operate / manage` 三档：读取请求要求 `view`，新增修改和业务动作要求 `operate`，删除要求 `manage`；管理员身份仍额外保护系统设置、用户管理和审计。内置模板包括管理员、单证人员、业务员和财务人员；管理员模板不可修改或删除，自定义模板被账号引用时不可删除。

账号与权限已经从系统设置分类中拆为独立的“系统维护 → 账号与权限”页面和 `/system/access-control` 路由，仅 `Full` 管理员可见；设置页不再重复放置同一入口。独立页面继续采用“账号管理 / 权限模板”两个清晰页签，避免在同一屏幕堆叠两套大型表单。角色使用系统管理员、单证人员、业务人员、财务人员等中文岗位名称和用途说明；账号状态使用明确的启用/停用标识，新增账号默认单证人员，密码字段区分新增必填与编辑留空不修改。权限模板继续使用“左侧模板目录 + 右侧分组模块权限”布局，支持新建、复制、保存、停用和删除；系统模板代码固定，管理员模板只读，模板代码收进高级信息。

账号开通策略固定为“首次 `admin` 登录初始化首个管理员，管理员再创建团队账号”，默认关闭公开自注册。该系统面向企业内部单证、销售和财务协作，公开注册会引入错误公司归属和权限模板误配；未来如果需要外部客户门户，应单独设计邀请制注册，不在当前内部工作区开放自由注册。PostgreSQL/Web/容器版空数据库第一次登录时，只有用户名 `admin` 可以使用本次输入的密码创建首个管理员，其他用户名不能先行初始化；该流程不增加部署环境变量，适合管理员部署后立即进入系统。普通账号和首个管理员初始/重置密码至少 8 个字符；密码使用 PBKDF2-SHA256、随机盐和定时安全比较。管理员不能停用或删除自己，不能删除最后一个启用管理员，已有业务数据归属的账号只能停用。

模板编辑器只显示普通管理员能理解的业务模块，不显示 `document.reference-data`、`common.product-reference`、候选项和单据输出等技术支撑能力，也不显示预留但未实现的 `finance.dashboard`。管理员只需选择业务模块与“仅查看 / 可操作 / 可管理”，服务端在保存和内置模板计算时自动补齐必要技术依赖，并使用最高显式权限避免降级；发票、商机和供应商会自动获得商品资料只读能力，发票另获得单位等单证基础资料读取，但不会因此开放主数据维护写接口或显示主数据导航。前端提交时同样只提交可见业务模块。SQLite 单机模式使用进程内 token 快照；PostgreSQL 多人模式只把 token 的 SHA-256 摘要写入数据库，每次请求重新读取当前启用账号及完整权限模板，因此停用账号、修改岗位或模板后不会继续沿用旧权限。管理员修改或删除账号后会主动撤销该账号全部会话。登录响应同时返回有效模块和访问级别，侧栏、默认入口、直达路由及付款、查询、OCR、邮件、CRM、商机、邮件模板和供应商页面的只读/操作控件使用同一权限；API 中间件仍独立执行强制授权。浏览器文件上传、服务器路径操作和 Tauri desktop token 属于运行环境权限，不能被普通模块模板自行放开。

权限模板和账号分配只存当前运行数据库，不写 `localStorage`、系统盘用户目录或额外 JSON 权限副本。项目尚未投产，本轮直接采用新数据库模型；已有开发期 SQLite 数据库若缺少权限模板表，应备份需要的样例数据后重建，不增加双 schema、运行期猜表或旧库兼容胶水。

### 1.4 多用户报表模板（2026-07-15 第一阶段已实现）

报表设计与“程序根默认模板”分为两条存储边界：默认/管理员文件模板继续位于程序根 `Templates/`，用户在 Web/容器版新设计的模板写入当前运行数据库 `UserReportTemplates` 表。普通用户只需拥有 `document.reports=operate` 即可新建、编辑、共享和删除自己的模板；`manage` 仍用于管理员文件模板、模板包和存储目录维护。

- 新模板默认私有；用户可选择“仅自己可见 / 同部门可见 / 同公司可见 / 团队成员可见”。
- 他人共享模板只读，可“复制为我的模板”后再修改，不会覆盖原作者内容。
- 模板所有者和管理员可发布/取消团队共享、停用或重新启用模板；停用模板继续保留历史和审计，但不参与预览、PDF 或正式输出。
- 报表类型是强隔离字段：`ExportDocument`（报关单证）与 `PaymentVoucher`（付款/报销）分别列表、复制和校验；服务端拒绝 `Invoice/Customer/Exporter/item` 与 `Payment.*` 交叉字段。
- 从默认模板创建时可传入同类型 `sourceTemplatePath`，由服务端读取程序根模板内容作为初始草稿；不允许跨类型复制。
- SQLite 单机版和 PostgreSQL 容器版都按用户所有权过滤普通账号；管理员可查看全部。模板内容、所属人、共享状态、版本号均只写当前运行数据库，不写系统 C 盘或浏览器持久化。
- 共享范围直接使用当前数据库模型的四值，不增加旧字段兼容、双写或运行期猜测分支；项目尚未投产，数据库按当前结构一次性定型。
- 模板及版本变更进入现有审计日志；`ContentHtml/BodyHtml` 只记录长度和 SHA-256 摘要，密码、密钥和令牌统一脱敏，避免审计库复制完整模板或敏感凭据。

具体接口、数据模型、交互规则和后续“数据库模板直接参与正式 PDF/打印渲染”工作见 `docs/报表模板多用户权限设计与实施记录.md`。

### 1.5 单一窗口权限闭环（2026-07-15 第二阶段已实现）

单一窗口页面统一使用 `document.single-window` 的 `view / operate / manage` 语义：列表、筛选、分页、详情和 GET 预检允许查看；草稿字段、默认值重建、修复、保存、提交包、客户端派发、回执收集/导入和目录档案保存要求 `operate`。COO、ACD 编辑器将可编辑字段集中在权限 `fieldset` 内，回执和预检结果保持只读；操作中心列表、批次详情、协作看板和回执包工作区均在只读时保留查询/打开路径能力并禁用写操作。COO/ACD 详情深链路由也要求 `operate`，避免通过地址栏进入可编辑页面。

参考词典继续属于系统设置管理员维护能力（`canManageSettings`），普通单一窗口用户可以查询和使用词典，不获得保存、恢复、JSON/Excel 导入或批量编辑权限；该边界有意独立于普通业务操作权限，避免把全局代码表维护误授予业务人员。

### 1.6 审计日志维护交互（2026-07-15 已实现）

审计日志管理明确分为两种动作：删除筛选结果只处理已提交筛选命中的记录，且至少需要一个筛选条件；按保留期清理忽略当前筛选，只删除早于保留天数的历史记录。两者都使用显示数量、范围和影响说明的确认弹窗，不再要求普通管理员输入 `DELETE`。导出按运行形态分为 Tauri 的 `save-to-path` 和浏览器/容器的 `download`：前者受 desktop token 保护并写入用户选择路径，后者返回标准 Excel attachment，不接收服务器路径。服务端使用 `confirmed` 布尔契约并拒绝无条件删除，前端维护区拆分为导出、保留期清理和折叠高级删除，保持桌面/浏览器响应式布局。

详细改动、验证证据和 Web/容器版受控浏览器下载验收边界见 `docs/审计日志维护交互重构与跨平台审查记录.md`。

### 1.7 全项目导出下载平台契约（2026-07-15 已实现）

全项目导出统一为两个平台适配方向：Tauri 使用原生保存对话框和受 desktop token 保护的 `save-to-path`；Web/容器使用标准 attachment，或对耗时输出采用“创建后台任务 -> 查询进度 -> 下载任务结果”。浏览器不再输入服务器绝对路径，上传文件只进入运行数据根 `Cache/BrowserUploads`，任务结果只进入运行数据根 `Exports/Browser`。普通 Web 任务响应只返回文件名，任务下载按用户归属授权；旧 `/export` URL、旧 operationId、旧 DTO 和兼容适配器已删除。

该契约已覆盖查询、审计日志、发票/付款 PDF、批量 ZIP、单据包、Excel 工具、PDF 合并、发票转移包、报表模板包、支持包和单一窗口包。CRM/供应商 Excel 保持既有 Blob 下载，WebDAV 云备份下载保持服务器备份管理语义。详细矩阵、路径、权限、缓存清理和验证证据见 `docs/全项目导出下载契约重构与多平台审查记录.md`。

### 1.8 商业化界面与装柜现场 PDF（2026-07-16 已实现）

普通业务页面不再直接展示程序根、`App_Data`、storage policy、服务器路径边界等开发审查文字；技术路径集中在管理员“运行检查”。维护工具拆为团队库、数据归属、运行检查和问题诊断四个任务页签，英文确认口令改为中文影响确认，服务端安全契约保持不变；响应式表单采用自动换列，普通窗口和小窗口不再因旧 owner、付款报销等字段产生页面级横向溢出。

装柜模拟 PDF 定型为现场作业单，只包含方案/柜型、装载摘要、现场提示、货物清单、伪 3D 效果图、俯视/侧视/柜门图和颜色图例；可旋转 Three.js 3D 与操作控件不进入 PDF。浏览器使用内存 Blob 下载，Tauri 使用原生保存对话框并由受限 PDF 写入命令保存用户显式路径，不新增数据库、服务器导出副本、系统临时目录或 C 盘默认落点。专项记录见 `docs/商业化界面与装柜现场PDF重构记录.md`。

### 1.9 SQLite 单机与 PostgreSQL 多人运行模式（2026-07-18 已加固）

数据库模式不再按“同一种部署随意切换”理解，而是固定为两条清晰产品边界：

- `SQLite` 只允许单机、单进程、回环地址使用。数据库文件位于当前运行数据根 `Database/`；连接启用外键、10 秒 busy timeout、连接池和 shared cache，初始化启用 `WAL + synchronous=NORMAL + foreign_keys=ON`。API 使用运行数据根 `Cache/Locks/sqlite-single-instance.lock` 持有排他租约，Tauri 桌面壳也启用单窗口插件；重复启动会恢复并聚焦现有窗口，不能让两个进程同时写同一数据库。它适合一名用户在一台电脑上使用，不允许把 `.db` 文件放到 NAS/共享文件夹后由多台电脑同时打开，也不允许网络监听时继续使用 SQLite。
- 单机版使用标准开源 SQLite，不使用 SQLCipher，也不再保留“密码已加密数据库”的伪逻辑。`.db` 文件本身是标准 SQLite 明文格式；需要静态数据保密的电脑应使用 BitLocker、FileVault、LUKS 或等效整盘/目录加密，并结合系统账号权限和备份介质加密。该选择保持依赖精简、跨平台一致和普通用户免安装额外数据库组件。
- `PostgreSQL` 用于局域网、Docker 和浏览器多人共享。用户会话写 `ApiUserSessions`，只保存不可逆 token 摘要；后台任务写 `ApiBackgroundJobs`，按单任务 upsert/删除，不再整表覆盖。API 重启后未完成任务会明确标记失败并保留可重试描述，已完成历史继续可查询。连接池默认上限为 30、最小预热 2、空闲连接 300 秒回收，可用 `EXPORTDOCMANAGER_DB_MAX_POOL_SIZE` 在 5–200 范围内按数据库规格调整，避免大量浏览器请求把 PostgreSQL 连接数无限放大。
- 数据库建表、结构检查和种子数据由进程级初始化协调器只成功执行一次；PostgreSQL 额外使用 advisory lock 防止并发初始化。认证中间件、持久会话签发/校验/撤销均走异步数据库调用，不在请求线程执行同步 PostgreSQL I/O。
- CRM 客户/联系人/跟进、商机、供应商/联系人/供货关系/评价、邮件模板、用户报表模板和付款记录均携带版本号或 RowVersion；前端保存时回传当前版本，服务端在数据库更新条件中校验并在冲突时返回 HTTP 409，避免多人编辑静默覆盖。批量状态更新也会递增版本，使已打开的旧编辑页必须刷新。
- 后台任务固定保存提交账号 ID；任务真正开始前再次读取当前启用账号和有效权限。账号在排队期间被停用或删除时，任务委托不会执行，并以明确失败原因结束。
- 单个 API 服务实例默认最多同时运行 4 个后台任务、每个用户 2 个、浏览器/PDF 重任务 2 个；可用 `EXPORTDOCMANAGER_JOB_GLOBAL_CONCURRENCY`、`EXPORTDOCMANAGER_JOB_USER_CONCURRENCY`、`EXPORTDOCMANAGER_JOB_BROWSER_CONCURRENCY` 按服务器资源调整。排队任务先取得用户/浏览器容量再占全局容量，避免某一用户的等待任务阻塞其他用户。
- 后台任务另有“运行并发”和“排队容量”两层门禁：全局默认最多保留 64 个未结束任务、单用户 12 个，可用 `EXPORTDOCMANAGER_JOB_GLOBAL_QUEUE_LIMIT`、`EXPORTDOCMANAGER_JOB_USER_QUEUE_LIMIT` 调整。超过上限立即返回 429 和普通用户可理解的等待提示，不再无限创建等待任务。用户任务信号量在该用户最后一个任务结束后删除，避免长期运行时按历史用户名持续增长。
- 任务历史默认保留 30 天、全局 2000 条、单用户 200 条，可用 `EXPORTDOCMANAGER_JOB_HISTORY_DAYS`、`EXPORTDOCMANAGER_JOB_HISTORY_LIMIT`、`EXPORTDOCMANAGER_JOB_USER_HISTORY_LIMIT` 调整；清理只处理已结束任务及其受控浏览器输出，不删除排队/运行任务。运行中进度仍实时保存在内存，但持久化最多约每秒一次，降低 SQLite JSON 整体重写和 PostgreSQL 高频 upsert 竞争。
- 受控 Playwright 宿主允许多个 BrowserContext 共用一个登记进程。单任务超时先关闭自己的 Page/Context；如同时还有其它页面，进程回收延迟到最后一个页面结束，避免一个用户超时中断另一个用户。宿主按次数/时间回收，并在启动时清理已退出 API 进程遗留的 `Cache/BrowserRuntime/automation-*`；PDF Chromium 仍为每任务独立进程，取消、超时和异常都终止进程树并重试清理临时目录。
- 管理员的数据归属改派已覆盖发票、付款、CRM 客户/跟进、供应商、商机、邮件模板、用户报表模板和装柜方案；停用账号不能成为目标，同一来源/目标会拒绝。大批量改派每 500 条保存并释放 EF 跟踪缓存，同时保持一个数据库事务和审计记录，避免一次性加载全部业务数据。
- 当前多人目标是“一套 PostgreSQL + 一个 API 服务实例 + 多个浏览器用户”，适合普通小团队和单台服务器。数据库记录已避免不同服务实例互相整表删除，但执行中取消令牌、任务领取和浏览器容量仍属于本 API 进程；在增加多个 API 副本前必须再实现数据库租约或队列领取，不能把当前版本直接声明为无状态横向扩展。
- 当前阶段不强制引入 Redis、Kafka、RabbitMQ 或商业数据库组件。若以后达到多 API 副本、数十个并行 PDF/OCR 任务或共享文件集群，再按实际负载增加 Redis/消息队列和 NAS/MinIO；不要为尚不存在的规模提前增加部署复杂度。

### 1.10 PP-OCRv6 Linux x64 与 Docker x64（2026-07-19 已接入）

- OCR 已统一为 `.NET IOcrService -> 长驻 Rust Sidecar -> PP-OCRv6 ONNX`。Sidecar 使用 `ort + image + ndarray` 完成解码、预处理、检测后处理和 CTC 解码；旧 C# OpenCvSharp OCR、原生运行包和运行时回退均已删除。
- 信用证 PDF 仅在无法提取有效文本时进行逐页栅格化并调用 Rust OCR，这属于 OCR 输入适配，不扩展为通用图片/PDF编辑模块。
- Excel批量结构分析优先使用随包 Rust `calamine` 分析器，.NET保留旧 `.xls` 和异常文件的兼容读取与业务映射；Rust只生成预览报告，不直接写数据库。
- Windows x64/ARM64、Linux x64/ARM64 与 macOS x64/ARM64 均构建对应 Rust Sidecar；发布包携带 PP-OCRv6 模型和 ONNX Runtime native，不安装 Python、OpenCV、PaddlePaddle Python 环境、系统 OCR 服务或商业 OCR SDK。
- 独立浏览器服务器包和 Docker 包在构建阶段执行 `--verify-ocr-runtime` 完成真实识别，失败则不生成发布包；可用 `EXPORTDOCMANAGER_OCR_RUNTIME=disabled` 显式关闭 OCR。
- 发布资源按 `Desktop / Server / Container` 配置生成：Rust模块共享程序根唯一ONNX Runtime；桌面和服务器只携带当前RID浏览器；Container使用发行版Chromium且不携带 `Browsers/`；Release包不包含PDB、`.lib`或未使用的ONNX托管/provider文件。
- 权限导航以API返回的 `moduleAccess` 为唯一现代契约：明确空权限不显示任何业务导航并进入无权限说明页；模块页面按view/operate/manage控制只读和写操作，后端中间件继续执行同级别强制授权，前端隐藏不能替代后端校验。
- Docker API 镜像在 Linux amd64 构建层执行同一真实 OCR 验证，并安装 `libgomp1`；运行时默认 `EXPORTDOCMANAGER_OCR_RUNTIME=enabled`。Linux arm64 镜像仍可构建其它功能，但明确跳过 OCR 验证并由运行时诊断返回架构不支持。
- PP-OCRv6 det/rec 模型与 PaddleOCR 为 Apache-2.0；ONNX Runtime 为 MIT；Rust依赖按各 crate 的 MIT/Apache-2.0 许可发布。具体来源记录随模型放在 `OcrModels/PaddleOCR/V6/THIRD_PARTY_NOTICES.md`。

## 2. 数据边界

- 原单证 `Customer`、`Invoice`、`Item`、报表 DTO、Scriban 字段和打印模板保持不变。
- 销售模块使用独立的 `CrmCustomer / CrmContact / CrmFollowUp`，不把销售字段堆入单证客户表。
- 供应链资料使用独立的 `SupplierCompany / SupplierContact / SupplierProductLink`；供货关系通过显式关联表连接现有产品，只读取产品编号和名称，不向产品表写供应商字段。
- `CrmCustomer.LinkedDocumentCustomerId` 只是可选桥接键；CRM 不反向修改单证客户。
- SQLite 未投产，数据库按当前模型直接重建，不增加旧结构兼容、双读双写或历史迁移胶水。
- SQLite 只作为单机单进程数据库；多人 Web/容器部署必须使用 PostgreSQL，不能通过共享 `.db` 文件绕过该边界。
- 项目尚未正式投产时，API、生成客户端、前端模型和运行配置统一以当前代码为唯一契约；字段调整直接同步修改并重新生成，不保留旧 sidecar、旧前端状态或旧配置猜测分支。
- PostgreSQL/局域网模式沿用现有用户归属范围；普通账号默认只能访问本人 CRM 数据，管理员可查看全部；会话和后台任务状态也进入共享数据库，不依赖某个浏览器或系统盘用户目录。

## 4. 运行与依赖原则

- 数据库、日志、缓存、WebView、备份和业务文件继续放在程序运行数据根或用户显式目录，不默认写系统 C 盘。Tauri 在运行数据根解析成功后，启动失败与 panic 诊断也优先写该数据根 `Logs/`；只有数据根尚未解析或日志目录不可写时才尝试程序根 `logs/` 诊断例外，全部候选不可写时直接在错误弹窗说明未生成日志，不再显示虚假文件路径。
- 程序根 `Templates/Resources/Browsers/Tools/OcrModels` 路径只做解析和存在性诊断，不因健康检查、设置页浏览或依赖发现自动创建；`Templates/` 是唯一报表模板根，内置模板、新建模板、编辑/重命名/删除和模板包导入继续统一使用 `Templates/Export` 或 `Templates/Internal`，不增加第二套模板目录。管理员可在设置页显式执行可写性检查；检查只在 `Templates/` 创建并立即删除短生命周期探针。API `/healthz` 通过结构化 `runtimePaths` 与 `runtimeDependencies` 统一说明程序资源、运行数据、当前数据库文件，以及浏览器、OCR、PostgreSQL 工具的真实就绪状态，并按 `core / feature / optional` 区分核心必需路径、按功能需要资源和可选组件。依赖检查只读取程序目录，不启动外部工具、不自动下载；前端只对核心或必需功能缺失显示故障，可选依赖未安装不得误报为软件损坏。
- 构建缓存继续使用仓库 `.codex-runtime` 与 `artifacts`；Rust/Cargo 路径由脚本执行非系统盘校验。
- .NET CLI、NuGet、npm 与构建 `TEMP/TMP` 统一优先进入仓库 `.codex-runtime/`；Tauri bundle 不再静默使用系统用户 `~/.cargo`，而是读取显式 `CARGO_HOME` 或从 `PATH` 中的 Cargo 推导工具链根，Windows 推导到系统盘时明确失败。开发机已安装的 `dotnet.exe`、系统 `curl.exe` 可作为工具入口，但缓存、临时文件、依赖下载和构建产物不随之写入系统盘用户目录。
- 三种桌面版本通过 `scripts/run-tauri-local.ps1 -ProductEdition Document|Sales|Full` 构建；Windows 便携目录脚本按版本输出独立目录。
- 局域网/容器 API 默认仍只允许回环监听；只有显式启用 `EXPORTDOCMANAGER_NETWORK_MODE=true` 且数据库为完整 PostgreSQL 配置时才允许 `0.0.0.0` 等网络监听。跨域前端必须进入 `EXPORTDOCMANAGER_ALLOWED_ORIGINS` 白名单；正式容器部署优先使用同源 Nginx 代理，不直接公开 API 端口。容器数据库镜像固定 `postgres:18-bookworm`，使用 Debian/glibc 多架构镜像；初始化显式选择 PostgreSQL 内置 `PG_UNICODE_FAST` Unicode locale，避免默认排序依赖 glibc/musl。PostgreSQL 18 数据卷挂载到 `/var/lib/postgresql`，跨大版本升级必须先完成备份恢复演练。
- 不新增 Python、Go、Next.js、Valkey、Meilisearch 或第二套数据库运行栈。
- 新增 UI 依赖必须优先选择免费开源、跨平台、无云服务绑定且仍在维护的库；现有 React、Lucide 和项目组件可以满足时不为视觉效果重复引入依赖。
- GitHub 公开仓库只包含客户端公钥验签，不包含 `apps/license-keygen-tauri/`、签发私钥、`KEY/` 内部工具或注册机产物。Windows 客户程序默认不构建注册机，私有源码存在且显式传入 `-IncludeLicenseKeygen` 时才生成内部工具。公开源码在本机脚本和 GitHub Actions 两层执行敏感路径、签发材料和 95 MiB 大文件门禁。
- Docker 公开发布使用 GitHub Actions 构建 API/Web 的 Linux amd64/arm64 镜像并保存到 GHCR；另提供 Windows/Linux 非 Docker 浏览器服务器包，由单个 ASP.NET Core 进程同源托管 React 与 API，连接部署机原生 PostgreSQL。GitHub Pages 不能运行 ASP.NET Core/PostgreSQL，不视为完整网页版托管。容器内 Chromium 来自 Debian 开源软件包并显式使用 `/usr/bin/chromium`；非 Docker 服务器包在构建期下载当前平台 Chrome Headless Shell 并打入发布包，源码仓库不提交数百 MB 浏览器副本。
- 产品版本、岗位和管理员能力必须在导航过滤、设置分类、前端路由和 API 中间件四层对齐；新增功能先声明所属 `document / sales / common / full-admin` 能力，再接入页面和端点，禁止仅靠侧栏隐藏形成表面权限。
- 跨工作区只读基础资料必须通过隐藏技术模块声明依赖，不能为了读取商品等选项直接授予整个主数据维护模块。当前商品读取统一使用 `common.product-reference`，发票、商机和供应商按需自动获得 `view`；商品新增、修改、删除仍只属于 `document.master-data`。
- 设置中心采用“状态协调壳 + 分类级独立组件 + 按需加载”结构：主页面只管理统一读取、保存、校验、消息和分类导航，运行/数据库、模板、Excel 导入、汇率、邮件/备份、AI/单一窗口和维护工具分别维护自己的展示与交互；进入分类前不加载对应代码，也不提前查询模板、签证机构或运行诊断数据。账号与权限是独立管理员工作区，不再嵌入设置分类。
- OpenAPI 以当前未投产契约为唯一事实源，文档工厂只负责顶层信息、服务器、安全方案、路径/Schema 合并和共享构造 helper；路径与 Schema 按系统、工具、单据、单一窗口、销售、主数据及公共模型分区维护，不在单个巨型工厂中继续追加，也不为旧客户端保留重复 Schema 或适配目录。
- Excel 发票导入采用“导入协调器 + 发票头 + 客户/出口商 + 主体块识别 + 普通商品明细 + 明细布局分析 + 订舱小表 + 字段查找/文本归一化 + 工作簿读取适配器”结构；协调器只负责分析器调用、工作表选择、取消和结果组装，`.xlsx`/`.xls` 读取、真实业务表头识别与各业务对象解析分别维护。字段类型校验与保守派生独立于单元格位置：目的国必须是可识别国家文本，不能接受金额；缺少独立目的国时只从明确国家名称的目的地或客户地址推断。导入草稿进入发票编辑器后由前端纯模型统一把 ASCII 小写字母转为大写，并提供相同规则的手动按钮；中文内容和数值不改。新增格式应扩展对应解析域和回归样本，不得重新把标签、主体、商品、订舱表和两种工作簿实现堆回单一服务文件。
- 单一窗口大型编辑器遵循“页面协调壳 + 专项编辑组件 + 纯业务规则模型”结构：页面只负责查询、mutation、草稿状态、撤销和跨组件编排，商品/附件/第三方企业等复杂工作区独立维护，条件显示、保存归一化、描述生成和代码转换保持可独立测试；不得重新把 JSX、文件交互与纯规则堆回单一页面文件。
- 单一窗口操作中心采用“极薄路由出口 + 独立导航 + 批次列表页 + 协作页 + 批次详情/客户端桥 + 回执包工作区 + 结果展示 + 通用表格 + 状态/路径模型”结构；COO、ACD 和参考词典只依赖轻量导航，不反向加载整个操作中心。提交包、客户端 OutBox/SentBox/InBox/FailBox、回执收集/导入和批次记录分别维护，新增操作中心动作不得重新堆回统一页面文件。
- 单一窗口参考词典采用“页面协调器 + 纯目录/校验/批量编辑模型 + 摘要展示”结构。六类词典定义、别名归一化、批量粘贴、重复校验、去重合并与 Excel 映射请求由纯模型统一维护，页面只协调 API、未保存保护和表格交互；模型可在 Windows/Linux/macOS 的 Node 构建环境独立回归，不创建本地词典副本或浏览器持久化。
- 报表设计器属性系统采用“主协调器 + 类型专项属性编辑器 + 明细表子域编辑器 + 共享控件 + 纯归一化模型”结构：主面板只负责当前页面、版区和组件类型的选择编排，页面/打印、网格、行、条件、图片和明细表分别维护；明细表继续按版式分页、分组小计、列内容、表尾汇总拆分。所有归一化与空单元格构造集中在纯模型，字段路径、列宽、边框和文字样式使用共享控件，不得把类型分支、列编辑和归一化规则重新堆回单一属性面板。
- 报表模板工作区采用“页面状态协调壳 + 查询/业务 mutation hook + 选择同步/选择动作 hook + 纯选择模型 + 纯派生状态模型 + 模板选择面板 + 工作区头部 + 用户/管理员/模板包面板 + 新版/源码编辑工作区 + 独立预览工作区 + 反馈组件”结构。预览 HTML、busy 聚合、内容有效性和全部操作可用性由纯派生函数计算；类型/模板/预览源切换动作独立维护；新版设计器和源码编辑器共享独立工作区。主协调页只保留状态所有权、成功回调和跨域流程编排，文件保持 `999` 行。后续新增功能应扩展对应模型、hook 或面板，不重新向协调页堆叠业务分支。
- 基础资料工作区采用“路由壳 + 通用列表 + 通用编辑器 + 字段控件 + 实体配置目录 + 纯辅助模型 + HS 专项工具”结构：客户、出口商、收款对象、商品、港口、单位和 HS 编码共享一致的搜索、分页、编辑、删除和键盘操作，不为每个实体复制页面；商品/单位辅助、HS 联网解析与清空保护保持独立，新增基础资料实体应优先扩展配置目录和必要的专项组件，不得重新把全部 CRUD、字段和工具堆入单一页面。
- HS 编码专项采用“本地有效性库 + 智能 Excel 预检/提交 + 联网参考查询”三段边界。HS Excel 只进入 HS 专项分析器，不复用或修改商业发票 Excel 分析链；增量资料不得因文件缺失停用本地编码，只有用户明确选择完整年度库才可标记疑似作废。历史商业发票保留原编码，本地商品候选只使用有效编码。联网 Provider 的静态解析优先按可信详情链接选择结果表，并兼容 i5a6 当前 `hscasefind` 申报实例结构；静态无结果或详情失败才进入受控浏览器降级。前端列表、导入向导和联网结果按任务分离，不在一个页面同时铺开完整表单、差异表和远程结果。
- HS Excel语义覆盖中国第三方税则常见变体：`规格型号`可作为申报要素、`许可证代码`作为监管条件，普通/优惠或最惠国/出口/消费/增值税率与出口退税率分别存储；同一Excel列只允许映射一个标准字段。新增供应商格式应以真实工作簿预检结果补充别名和回归样本，不把特定文件列号硬编码进导入器。
- 浏览器能力采用“共享治理、进程隔离”：`BrowserRuntimeManager / BrowserExecutableResolver` 统一负责运行目录浏览器定位、全局资源预算、PID归属、取消超时和退出清理；PDF使用每任务独立命令行Chromium，HS动态网页降级使用受控Chromium + Playwright CDP宿主。两类业务不共享Browser进程或页面状态，避免PDF打印参数与网页自动化会话互相污染。新增浏览器业务必须接入运行时管理器，不得自行扫描/结束Chrome进程。
- 面向普通用户的销售页面遵循“单页单主要任务”：目录、编辑、导入导出、关联、预览和历史通过清晰分区切换，不在一个屏幕同时铺开全部功能；供应产品等子模块内部也必须把关系目录和新增/编辑表单分开，不能因为已经位于上层任务页就再次把列表与完整表单堆在一起。
- 销售工作区任务分区使用统一页签语义和共享视觉规则；普通窗口保持双列表单，小于 `860px` 自动改为单列，小于 `560px` 时操作按钮纵向铺满，任务页签横向滚动而不把所有功能挤入同一屏幕。页签支持键盘方向键、Home 和 End 切换。
- 销售概览只在客户、联系人、待跟进和进行中商机均为空时显示三步首次使用引导；已有业务数据后自动回到统计和待办视图。概览指标可直达对应任务，客户与商机页面通过轻量 `?view=` 参数打开指定分区，不保存额外用户状态。
- 商业化界面优先使用一致的状态语言、主次按钮、搜索分页、确认提示和空状态；业务状态统一按积极、跟进提醒、风险、停用和普通信息显示，保存成功、输入提醒与接口错误使用不同反馈语义；目录中的客户、商机、供应商、产品和邮件主题等长文本保持紧凑单行并提供完整悬停提示，避免普通窗口被异常数据撑宽；目录筛选不得导致刚保存的记录从编辑上下文中消失，必要时清除会隐藏新记录的筛选并保持当前记录选中；高级能力逐步显示，不要求普通用户理解内部模块或数据库结构。

## HS 编码知识中心（2026-07-20）

HS 模块采用“当前年度标准编码 + 历史申报实例 + 替代关系 + 用户反馈”四层模型。标准编码用于判断当前可用性；申报实例保存普通商品名称和真实规格；替代关系处理年度变更；反馈只调整本机推荐。联网 Provider 是可替换的补充来源，不是本地查询的强依赖。单机知识可通过 `.edmhs` 在两台电脑间迁移，且不携带商业发票等业务数据。

前端入口为 `/master-data/hs-knowledge/:section`，侧栏组名为“知识中心库”、子菜单为“HS 编码库”，任务分为 `search / examples / history / annual / transfer / online`；原 `/master-data/hs-codes` 保持标准编码列表和人工编辑职责。知识中心路由不会再同时激活“主数据维护”。
知识中心页面不重复渲染应用工作区顶部已经提供的英文眉题、中文标题、总说明和返回按钮，内容区直接从任务标签开始。六个任务标签在非选中状态也使用清晰边框和实体底色，避免把一级任务导航表现成弱化文字链接；年度税则和联网补充的任务提示采用收窄短说明，不横跨整张工作卡片。
历史资料学习属于“候选发现层”，只读取 `Products / Items / CustomsCooItems` 并返回待审核视图，不直接写 `HsCodeDeclarationExamples`。人工确认是候选进入正式知识层的唯一入口，确保系统推测不会被当作事实。
“同义词”与“相关词”分开处理：全棉/纯棉等可安全归一的表达进入同义词层；针织/钩编等相关但不等价的工艺只参与相关度加分，不改写知识事实。
联网 Provider 输出先进入 `HsCodeRemoteCandidates` 候选层；`HsCodeDeclarationExamples` 只保存人工确认实例。候选层不能被搜索服务当作正式证据，审核服务是两层之间唯一写入通道。
Linux ARM64浏览器运行时采用随包 `Browsers/ChromiumArm64`，来源和产品标识与Google官方Chrome Headless Shell分离。浏览器解析器不关心发行品牌，只通过统一可执行文件契约向PDF和网页自动化提供能力。

### i5a6 网页证据边界（2026-07-21 审查补充）

i5a6 搜索页存在三种不可混淆的结构：`resultfind` 后的标准编码结果表、`hscasefind`/“申报实例查询结果”直接实例表，以及标准编码详情页的 `#sbsl` 申报实例汇总表。标准编码结果行中的“实例汇总”链接通常指向 `/hscode/detail/{code}#sbsl`；详情页上部还包含申报要素、单位、税率、监管条件、检验检疫、CIQ 和分类层级，`#sbsl` 首屏通常只展示约 20 条代表实例，链接显示的“534条”等是汇总数量而不是可直接遍历的全部行。

联网 Provider 现已返回类型化证据：`StandardCode`、`DeclarationExample` 和 `ReplacementEvidence` 分开传递，并保留 `summaryUrl / instanceCount / evidenceUrl / observedAt`。详情解析覆盖 `#hscode-detail`、`#sbsl`、个人行邮税号、CIQ 和分类层级；旧实例的商品名称/规格仍可作为历史语义样本，但旧编码推荐只产生待审核映射。网页观察时间不会写成实例年份，i5a6 标准行也不会自动写成当前年度有效税则。

联网查询明确采用两条数据流：上方结果表只展示按编码去重的非作废 `StandardCode`；最初搜索页的全部 `DeclarationExample` 按“原始编码 + 商品名称 + 商品规格”整行去重后进入 `HsCodeRemoteCandidates`。当“男T恤”等搜索页只有历史实例、没有当前标准行时，协调服务按每个唯一旧编码访问一次详情页，读取“已作废 / 推荐查询”，再查询推荐前缀并合并当前标准行；自动详情访问全请求最多 12 个唯一编码、推荐递归最多 3 层，单个失败不影响其它编码。详情页 `#sbsl` 和推荐查询页产生的额外实例不会自动混入本次直接候选，避免证据范围无控制扩张。无法解析当前标准编码时，上方保持为空并明确提示候选已保存，不再用旧实例伪装标准结果。

本地有效税则选择器对至少 4 位的纯数字条件采用 `NormalizedCode.StartsWith` 前缀语义；例如 `6109` 只匹配 `6109100000` 等以 `6109` 开头的编码，不匹配 `2846109010` 这类中间包含值。商品名称等文本条件继续使用名称/描述检索，两个语义不互相污染。

解析层继续采用跨平台 `.NET 8 + HtmlAgilityPack`，动态 DOM 使用统一 `.NET 8 + Microsoft.Playwright` 浏览器宿主降级。解析器按表头语义、字段标签和结构质量评分选择区域，不依赖单一 `id/class/XPath`；已覆盖包裹层、表头行、列顺序和移动卡片类名变化。该任务属于网络 I/O 和语义校验，不存在值得引入 Rust Sidecar 的已证实性能瓶颈；Rust 仍用于 Excel 结构分析和 OCR，避免为同一 API 增加第二语言进程边界。

知识中心已经落实当前年度有效性门禁：只有本地 `Active` 年度税则编码可由人工确认进入正式实例；第三方详情补全不自动写主库，显式保存也标记为 `ReferenceOnly`，后续年度税则导入才可升级为 `Active`。智能查询增加性别、材质、织造方式和品类冲突惩罚，界面显示匹配理由/冲突并把分数明确命名为“文本匹配分”。本模块定位仍是“可审查的辅助检索”，不能作为无人审核的自动报关归类决策器。

### 可信 Active、发票回填与审核生命周期（2026-07-21）

`Active` 不再只是字符串状态，而是必须同时满足“可信来源 + 2000—2100 适用年度 + 最近验证时间”的正式税则事实。手工新建 `HsCode` 默认 `ReferenceOnly`，普通编辑不能把参考或作废编码提升为 `Active`；建立和更新正式有效编码只允许走年度税则导入或经过完整性校验的 `.edmhs` 迁移。智能查询、历史确认、联网候选确认、反馈确认和发票一键回填共同使用同一 `HsCodeValidityPolicy`，避免各入口出现不同有效性口径。

发票商品明细新增“智能 HS”侧边面板：查询词由当前行中文/英文品名、材质和品牌组合，结果展示税则来源、年度、验证时间、文本匹配分、正式实例数和属性冲突；只有可信 `Active` 可一键回填当前行。回填编码时可补齐空白中文单位和退税率，并记录本次人工确认反馈，形成“年度税则 -> 正式实例 -> 发票使用 -> 反馈学习”的闭环。面板使用 `/api/invoices/hs-*` 发票专用只读/反馈别名并由 `document.invoices=operate` 授权，不要求额外显示“主数据维护”，也不把主数据新增、编辑或删除权限隐式授给发票用户。

商品主数据编辑器不再预载固定 200 条 HS 编码。用户输入至少 4 位编码后采用 250ms 防抖向服务端查询前 20 条，结果缓存 1 分钟；失焦确认时再做精确校验，只套用完全一致且可信的 `Active`。因此浏览器内存和首次加载时间不再随年度税则总量线性增长，大型税则库仍通过数据库分页、数字前缀查询和服务端过滤工作。

申报实例库和联网候选池都采用服务端分页。实例库支持选择和管理级批量删除；候选池保留待审核、已确认、已忽略三类审核历史，支持批量确认、批量忽略和恢复待审核。恢复已确认候选时同步删除其 `RemoteConfirmed:*` 正式实例，保证“撤销审核”不会留下继续参与推荐的隐性知识。`view` 只显示查询和历史，`operate` 显示新增、编辑、确认、忽略和撤销，删除只对 `manage` 显示；API 中间件仍按同一档位强制授权。

远程联网服务只保留 `IHsCodeRemoteProvider` 编排。`HsCodeService` 不再拥有第二套 i5a6 HttpClient、HTML 表格和详情解析；当前 `I5a6HsCodeProvider + I5a6PageParser` 负责来源特有行为，协调服务只负责 Provider 优先级、证据合并、旧编码推荐追踪和故障降级。新增数据源应实现 Provider，不修改知识服务、发票面板或候选审核流程。

## 权限、业务归属与正式多人运行边界（2026-07-21）

权限采用前后端双层但不互相替代的模型。React 根据会话返回的 `moduleAccess` 控制工作区、路由和按钮显隐，公共业务组件的 `canOperate / canManage` 等参数必须显式传入，缺少参数会在 TypeScript 编译阶段暴露；API 中间件依据模块和 HTTP 方法再次执行 `view / operate / manage` 校验，端点内部再处理使用 POST 表达的批量删除等特殊管理动作。因此隐藏按钮只改善普通用户体验，真正的越权防护始终位于服务端。

PostgreSQL 多用户模式下，发票、付款、CRM、供应商、商机、用户模板、后台任务和装柜方案均有明确账号归属或作用域。装柜方案使用 `OwnerUserId / DepartmentId / CompanyScope` 隔离普通账号，并用 `VersionNumber` 防止同一记录在多个浏览器会话中被旧版本覆盖；管理员保留全局审查能力。删除账号前必须确认其不再拥有主要业务实体，否则只能停用。共享型参考数据（例如柜型和标准主数据）继续由模块权限控制，不伪装成个人数据。

当前可承诺的正式多人拓扑是“一套 PostgreSQL、一个 API 服务实例、多个浏览器用户”。SQLite 仅面向单机、单进程并绑定回环地址，不支持把数据库文件放到 NAS 后供多台电脑并发打开。多个 API 副本仍需要数据库任务租约、跨实例取消、共享文件存储和压力测试后才能进入支持范围；Docker/PostgreSQL 多岗位压力、备份恢复、HTTPS 反向代理、大文件/PDF/OCR 容量以及真实 Windows/macOS/Linux 设备与签名更新链也仍需生产前实机验收。

前端继续采用 React、TypeScript、TanStack Query、React Router、Lucide、Three.js、html2canvas、jsPDF 和 Vite 等免费开源库。页面已具备统一导航、权限态、空/忙/错状态、分页和窄屏适配，生产构建按路由拆包；后续视觉和维护性优化优先拆分大型业务协调页和全局样式表，而不是引入重量级商业组件。当前 NuGet 与 npm 全依赖审计均无已知漏洞。
## 2026-07-22 多用户共享主数据并发边界

- 客户、出口商、商品、收款对象、港口、单位和 HS 编码统一采用 API `rowVersion` 乐观并发令牌。读取详情时返回令牌，保存时携带原令牌；数据库写入成功后由审计拦截器生成新令牌。
- 两个浏览器同时编辑同一条共享资料时，先保存者成功，后保存的旧页面返回 HTTP 409，并提示加载最新数据，不再使用“最后保存者覆盖前者”的隐式策略。
- HS 编码普通编辑受版本令牌保护；年度税则导入和联网候选写入仍走各自的批处理/审核边界。批量任务不会伪装成用户打开的旧编辑页面，也不会把第三方参考资料直接升级为当前有效编码。
- 权限边界仍为“前端显隐与禁用 + 路由守卫 + API 最终校验”：`view` 只读、`operate` 新建/编辑/执行、`manage` 删除和批量破坏性操作；管理员系统页另受管理员身份、产品版本和桌面运行环境约束。
- 当前正式多人拓扑仍限定为“一个 PostgreSQL 数据库 + 一个 API 实例 + 多个浏览器用户”。本轮并发令牌解决同一 API 下的共享资料编辑冲突，但不等同于多 API 副本能力；横向扩容仍需数据库任务租约、跨实例取消和共享文件存储。
