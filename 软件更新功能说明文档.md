# 软件更新功能说明文档
> 适用项目: `ExportDocManager_CS`
>
> 适用对象: 第一次接触“软件自动更新”的新手、小白、后续维护人员
>
> 最近整理日期: 2026-04-15

## 1. 先看结论

这个软件现在的更新功能，已经支持下面几种更新源:

- 本机本地目录
- 局域网共享目录 / NAS
- 独立网络服务器
- GitHub

你不一定要买服务器，也不一定非要会网站开发。

如果你只是自己用，或者给办公室几台电脑用，最简单的方案通常是:

- 单机自己用: 本地目录
- 公司内网几台电脑用: 局域网共享目录
- 给外网用户用: 独立服务器 或 GitHub

软件内部只需要配置一个“更新清单地址”，也就是 `update.json` 的位置。
软件会先读取这个清单，再根据清单去下载新版压缩包。

## 2. 这个更新功能是怎么工作的

软件更新的流程，尽量用大白话说明如下:

1. 软件启动时，或者你手动点击“检查更新”。
2. 软件读取你设置的 `update.json`。
3. 软件比较“当前版本”和“清单里的最新版本”。
4. 如果发现有新版本，就下载 zip 更新包。
5. 如果清单里写了 `sha256`，软件还会先校验更新包有没有损坏。
6. 更新包不会直接覆盖正在运行的软件目录。
7. 软件会先把更新包下载到临时目录，再解压到 staging 临时目录。
8. 当软件退出后，外部更新脚本才会把新文件复制到正式目录。
9. 复制完成后，软件自动重新启动。

这样做的好处是:

- 更稳定
- 不容易把正在运行的程序覆盖坏
- 更新失败时，不容易把原程序弄残
- 更适合局域网、服务器、GitHub 多种部署方式

## 3. 你需要准备什么

无论你选哪种更新方式，核心都只需要两个东西:

- 一个 `update.json`
- 一个新版程序的 zip 压缩包

推荐目录结构:

```text
更新目录/
├─ update.json
└─ packages/
   └─ ExportDocManager_1.2.0.zip
```

最推荐这样做的原因是:

- 容易看懂
- 版本文件集中
- 后面发新版本时不容易乱
- 相对路径好写

## 4. 软件里要怎么设置

### 4.1 在软件界面里设置

打开软件后:

1. 进入“系统设置”
2. 找到“更新清单地址”
3. 填入 `update.json` 的地址
4. 保存设置

填写示例:

- 本地目录:
  - `D:\ExportDocManagerUpdate\update.json`
- 局域网共享目录:
  - `\\NAS\ExportDocManager\update.json`
- 独立服务器:
  - `https://updates.example.com/edm/update.json`
- GitHub Pages:
  - `https://yourname.github.io/edm-update/update.json`

设置完成后:

- 软件启动时会自动检查更新
- 你也可以在主界面工具菜单里手动点“检查更新”

### 4.2 如果软件打不开，也可以手工改配置

软件配置保存在程序目录下的 `appsettings.json`。

如果要手工改，可以找到类似下面的位置:

```json
{
  "System": {
    "UpdateUrl": "\\\\NAS\\ExportDocManager\\update.json"
  }
}
```

注意:

- Windows 路径里的反斜杠在 JSON 里要写成 `\\`
- 改完后保存，再重新打开软件

## 5. `update.json` 是什么

`update.json` 就是一张“更新说明单”。

软件不会凭空知道哪里有新版本，它只会先看这张说明单。

### 5.1 最简单的示例

```json
{
  "version": "1.2.0",
  "downloadUrl": "packages/ExportDocManager_1.2.0.zip",
  "releaseNotes": "1. 修复若干Bug\n2. 优化更新流程",
  "isMandatory": false
}
```

### 5.2 推荐示例

```json
{
  "version": "1.2.0",
  "downloadUrl": "packages/ExportDocManager_1.2.0.zip",
  "sha256": "这里填写ZIP文件的SHA256",
  "releaseNotes": "1. 修复若干Bug\n2. 优化OCR资源占用\n3. 提高更新稳定性",
  "isMandatory": false,
  "releasePageUrl": "https://example.com/releases/1.2.0"
}
```

### 5.3 每个字段是什么意思

- `version`
  - 最新版本号
  - 只要它比当前软件版本高，软件就会认为有更新
- `downloadUrl`
  - 更新包地址
  - 可以写相对路径、绝对本地路径、共享路径、HTTP/HTTPS 地址
- `sha256`
  - 更新包哈希校验值
  - 建议填写
  - 有它就能在安装前判断 zip 有没有损坏
- `releaseNotes`
  - 更新说明
  - 软件发现新版本时可以显示给用户看
- `isMandatory`
  - 是否强制更新
  - `true` 表示用户不能“跳过更新继续用”，只能更新或退出
- `releasePageUrl`
  - 版本说明页面地址
  - 当前建议当作备用字段保留，填不填都可以

### 5.4 `downloadUrl` 怎么写最合适

最推荐写“相对路径”，比如:

```json
{
  "version": "1.2.0",
  "downloadUrl": "packages/ExportDocManager_1.2.0.zip"
}
```

这样有一个好处:

- 如果你以后把同一套更新文件从本地目录搬到局域网
- 再从局域网搬到服务器
- 再搬到 GitHub

通常只要 `update.json` 和 `packages` 的相对位置不变，`downloadUrl` 就不用改。

## 6. 版本号要怎么改

这个很重要。

如果你发了新包，但程序自己的版本号没改，软件可能会判断“当前已经是最新版”，结果不更新。

当前项目版本号主要看这里:

- `ExportDocManager/Properties/AssemblyInfo.cs`

例如:

```csharp
[assembly: AssemblyVersion("1.2.0.0")]
[assembly: AssemblyFileVersion("1.2.0.0")]
```

推荐做法:

- 每次发新版，都先把这里的版本号改掉
- 然后让 `update.json` 里的 `version` 和它保持一致

推荐统一格式:

- `1.2.0`
- 或 `1.2.0.0`

## 7. 更新包怎么做

### 7.1 最简单理解

更新包就是“新版程序文件夹”的 zip 压缩包。

### 7.2 推荐做法

建议你先得到一个完整可运行的新版本目录，再压缩。

目录里通常应该包含:

- `ExportDocManager.exe`
- 对应的 `dll`
- `Templates`
- `Resources`
- `Seals`
- `OcrModels`
- 其它运行所需文件

不要把这些东西打进更新包:

- `bin`
- `obj`
- `TestResults`
- `.vs`
- 临时日志
- 个人配置残留

### 7.3 压缩包内部建议长什么样

推荐压成这种结构:

```text
ExportDocManager_1.2.0.zip
└─ ExportDocManager/
   ├─ ExportDocManager.exe
   ├─ ExportDocManager.dll
   ├─ Templates/
   ├─ Resources/
   ├─ Seals/
   └─ OcrModels/
```

这样最清晰，也最不容易误操作。

当前软件也兼容下面这种结构:

```text
ExportDocManager_1.2.0.zip
├─ ExportDocManager.exe
├─ ExportDocManager.dll
├─ Templates/
├─ Resources/
├─ Seals/
└─ OcrModels/
```

但从管理角度来说，还是“外面包一层文件夹”更整齐。

### 7.4 怎么生成 SHA256

推荐你每次发布都生成一次 SHA256。

Windows 命令行:

```bat
certutil -hashfile ExportDocManager_1.2.0.zip SHA256
```

PowerShell:

```powershell
Get-FileHash .\ExportDocManager_1.2.0.zip -Algorithm SHA256
```

把得到的哈希值，复制到 `update.json` 的 `sha256` 字段里即可。

## 8. 第一次发布更新的完整流程

如果你是第一次做更新，推荐严格照下面做:

1. 改程序版本号
2. 编译或发布新版程序
3. 准备一个完整可运行的新版本文件夹
4. 压缩成 zip
5. 生成 zip 的 SHA256
6. 编写 `update.json`
7. 把 `update.json` 和 zip 放到更新目录
8. 在软件里设置“更新清单地址”
9. 手动点“检查更新”
10. 在测试机上先完整跑一遍更新流程
11. 确认没问题后，再给正式用户使用

## 9. 方案一: 本机本地目录架设

这是最简单的方式。

适合:

- 只有你自己一台电脑用
- 想先测试更新功能
- 暂时不想上服务器

### 9.1 目录示例

```text
D:\ExportDocManagerUpdate\
├─ update.json
└─ packages\
   └─ ExportDocManager_1.2.0.zip
```

### 9.2 软件里怎么填

填:

```text
D:\ExportDocManagerUpdate\update.json
```

### 9.3 优点

- 最简单
- 完全不用网络
- 非常适合先测试

### 9.4 缺点

- 只能这台电脑自己用
- 不适合多人共享

## 10. 方案二: 局域网共享目录 / NAS 架设

这是办公室、小团队最实用的方式之一。

适合:

- 公司内部几台电脑一起用
- 不想买公网服务器
- 有 NAS 或 Windows 文件共享

### 10.1 目录示例

假设你的共享目录是:

```text
\\NAS\ExportDocManager\
├─ update.json
└─ packages\
   ├─ ExportDocManager_1.2.0.zip
   └─ ExportDocManager_1.2.1.zip
```

### 10.2 软件里怎么填

填:

```text
\\NAS\ExportDocManager\update.json
```

### 10.3 共享权限建议

- 普通使用者:
  - 只给“读取权限”
- 维护更新的人:
  - 给“读取 + 写入权限”

这样更安全。

### 10.4 发布新版本怎么做

1. 把新的 zip 放到 `packages`
2. 修改 `update.json` 的版本号、下载文件名、哈希值
3. 保存
4. 用户端下次启动或手动检查更新时，就会发现新版

### 10.5 优点

- 非常适合办公室
- 不用公网
- 维护简单
- 用户端只要能访问共享目录即可

### 10.6 缺点

- 只能在同一局域网使用
- 如果共享目录断开，更新就会失败

### 10.7 给新手的建议

如果你现在是“小团队内部使用”，优先选这个方案。

它通常是:

- 成本最低
- 学习成本最低
- 出问题最好排查

## 11. 方案三: 独立网络服务器架设

这是给外网用户更新时更标准的做法。

适合:

- 用户不在同一个办公室
- 需要通过互联网更新
- 想做长期稳定更新服务

### 11.1 服务器本质上要做什么

其实很简单。

服务器只需要能提供静态文件下载即可，也就是能让用户访问:

- `https://你的域名/update.json`
- `https://你的域名/packages/xxx.zip`

你不一定要写后端程序。

很多时候只要:

- IIS
- Nginx
- Apache
- 宝塔面板静态站点

就够了。

### 11.2 目录示例

```text
/var/www/edm-update/
├─ update.json
└─ packages/
   └─ ExportDocManager_1.2.0.zip
```

### 11.3 软件里怎么填

填:

```text
https://updates.example.com/edm/update.json
```

### 11.4 `update.json` 示例

```json
{
  "version": "1.2.0",
  "downloadUrl": "packages/ExportDocManager_1.2.0.zip",
  "sha256": "填写SHA256",
  "releaseNotes": "修复若干问题"
}
```

因为这里用了相对路径，所以软件会自动理解成:

- 清单地址:
  - `https://updates.example.com/edm/update.json`
- 下载地址:
  - `https://updates.example.com/edm/packages/ExportDocManager_1.2.0.zip`

### 11.5 建议

- 外网场景优先使用 `HTTPS`
- 每个版本保留独立 zip
- 不要把 zip 文件名一直写死成同一个

推荐:

- `ExportDocManager_1.2.0.zip`
- `ExportDocManager_1.2.1.zip`
- `ExportDocManager_1.3.0.zip`

这样便于回滚和排查

### 11.6 优点

- 适合正式对外使用
- 访问快
- 可控性高
- 适合长期维护

### 11.7 缺点

- 需要服务器或虚拟主机
- 对新手来说，比局域网共享稍复杂一点

## 12. 方案四: GitHub 架设

GitHub 适合:

- 你已经在用 GitHub 管理项目
- 想减少自建服务器维护
- 用户分布在外网

这里推荐两种思路。

### 12.1 GitHub 方案 A: GitHub Pages 直接放静态文件

最容易理解。

做法是:

1. 建一个 GitHub 仓库
2. 开启 GitHub Pages
3. 把 `update.json` 和 `packages/*.zip` 一起放到 Pages 对应目录

例如:

```text
仓库根目录
├─ update.json
└─ packages/
   └─ ExportDocManager_1.2.0.zip
```

如果你的 GitHub Pages 地址是:

```text
https://yourname.github.io/edm-update/
```

那么软件里的“更新清单地址”就填:

```text
https://yourname.github.io/edm-update/update.json
```

`update.json` 里可以写:

```json
{
  "version": "1.2.0",
  "downloadUrl": "packages/ExportDocManager_1.2.0.zip",
  "sha256": "填写SHA256",
  "releaseNotes": "修复若干问题"
}
```

优点:

- 最直观
- `downloadUrl` 可以继续用相对路径
- 非常适合新手理解

缺点:

- 如果 zip 比较大，管理起来不如 Releases 舒服

### 12.2 GitHub 方案 B: `update.json` 放 Pages，zip 放 Releases

这是更推荐的 GitHub 方案。

思路是:

- `update.json` 放 GitHub Pages 或 raw 地址
- 程序 zip 放 GitHub Releases 附件

例如 `update.json`:

```json
{
  "version": "1.2.0",
  "downloadUrl": "https://github.com/yourname/edm-update/releases/download/v1.2.0/ExportDocManager_1.2.0.zip",
  "sha256": "填写SHA256",
  "releaseNotes": "修复若干问题"
}
```

这种方式的好处:

- 版本文件清晰
- Releases 专门适合放发布包
- 不用把大 zip 混在 Pages 目录里

### 12.3 GitHub 新手推荐怎么选

如果你是完全新手:

- 想先跑通:
  - 选 GitHub Pages 直接放静态文件
- 想后期更规范:
  - 选 `update.json` 放 Pages，zip 放 Releases

### 12.4 GitHub 方案注意事项

- 私有仓库不适合新手直接做更新源
  - 因为下载通常涉及权限和鉴权
- 如果用户是普通外部用户，建议用公开可访问的地址
- `downloadUrl` 如果指向 GitHub Releases，建议写完整绝对地址，不要写相对路径

### 12.5 GitHub 操作流程简版

#### 做法一: Pages 直接放静态文件

1. 新建仓库
2. 提交 `update.json`
3. 提交 `packages/xxx.zip`
4. 开启 GitHub Pages
5. 得到 Pages 地址
6. 把这个地址填到软件里

#### 做法二: Pages + Releases

1. 新建仓库
2. 开启 Pages
3. Pages 里只放 `update.json`
4. 到 GitHub Releases 新建版本
5. 上传 zip 附件
6. 把附件下载地址写进 `update.json`
7. 把 Pages 的 `update.json` 地址填到软件里

## 13. 每次发新版本应该怎么做

以后每次更新，按这个清单做最稳:

1. 修改程序版本号
2. 重新生成新版程序目录
3. 打包成 zip
4. 生成 SHA256
5. 上传 zip 到更新目录 / 服务器 / GitHub
6. 修改 `update.json`
7. 检查 `version`
8. 检查 `downloadUrl`
9. 检查 `sha256`
10. 检查更新说明
11. 用一台测试机先跑一遍
12. 没问题后再正式发布

## 15. 怎么测试更新有没有真的成功

推荐用“测试机”或“测试目录”先测，不要直接在正式环境第一次就上。

测试步骤:

1. 准备一个旧版本程序
2. 给它设置 `update.json` 地址
3. 保证 `update.json` 的版本号更高
4. 打开软件
5. 手动点“检查更新”
6. 确认看到更新提示
7. 点击更新
8. 等软件退出并自动重启
9. 检查程序是否变成新版本
10. 检查模板、资源、OCR 模型是否正常

重点检查:

- 能不能正常启动
- 数据会不会丢
- 模板能不能预览
- OCR 能不能正常识别
- 更新后“检查更新”会不会提示已经最新

## 16. 常见问题排查

### 16.1 软件一直提示“当前已是最新版本”

常见原因:

- 程序自身版本号没改
- `update.json` 里的 `version` 没比当前版本高
- 配置的不是正确的 `update.json`

先检查:

- `AssemblyInfo.cs` 版本号
- `update.json` 版本号
- 软件里“更新清单地址”有没有填错

### 16.2 发现新版本了，但下载失败

常见原因:

- `downloadUrl` 写错
- 共享目录没权限
- 服务器地址打不开
- GitHub 地址写错

先检查:

- 把 `downloadUrl` 直接复制到浏览器里，看能不能下载
- 如果是共享目录，看看用户电脑能不能访问

### 16.3 下载了，但更新安装失败

常见原因:

- zip 包内容不完整
- zip 包里根目录结构混乱
- 压缩包损坏
- 杀毒软件或权限拦截复制

建议:

- 重新生成 zip
- 重新算 SHA256
- 在测试机上先完整验证

### 16.4 `SHA256` 校验失败

说明:

- 下载的 zip 和你发布时的 zip 不一致
- 可能是文件损坏
- 也可能是你改了 zip，但忘了更新 `sha256`

解决:

1. 对最终发布出去的 zip 重新计算 SHA256
2. 把新值写入 `update.json`
3. 再试一次

### 16.5 局域网共享能打开，但软件更新还是失败

重点检查:

- 共享路径是不是写成了文件夹，而不是 `update.json`
- 用户有没有读取权限
- 网络有没有断
- 路径里是不是实际不存在那个 zip 文件

正确示例:

```text
\\NAS\ExportDocManager\update.json
```

不是:

```text
\\NAS\ExportDocManager\
```

### 16.6 GitHub 可以打开页面，但软件下载不了

常见原因:

- `downloadUrl` 不是最终可下载地址
- 填成了网页地址，不是附件下载地址
- 私有仓库用户没权限

建议:

- GitHub Releases 方案里，直接复制“附件下载链接”
- 新手尽量先用公开仓库测试

## 17. 安全和稳定性建议

为了让更新更稳定，建议长期遵守下面这些规则:

- 每次发布都保留旧版本 zip，不要立刻删
- `update.json` 先在测试环境验证，再给正式用户
- 每次发布都生成 SHA256
- 外网更新尽量使用 HTTPS
- 不要让普通用户有更新目录写权限
- 不要把开发中的半成品 zip 当正式包发布
- 不要把 `bin/obj/TestResults` 打进更新包

## 18. 我最推荐的实际落地方案

如果你问我“最省心、最适合你现在这个项目”的方案，我会这样建议:

### 18.1 如果主要是自己和办公室内部使用

首选:

- 局域网共享目录 / NAS

原因:

- 最简单
- 最便宜
- 最容易维护
- 软件已经原生支持共享路径

### 18.2 如果以后可能有外网用户

首选:

- 独立静态服务器

备选:

- GitHub Pages + Releases

原因:

- 外网更稳定
- 更方便长期管理多个版本
- 便于后续做更规范的发布流程
