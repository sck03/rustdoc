## 2. 当前真实模块地图

从当前代码看，项目主干已经从旧 Windows 桌面单体收敛为跨平台后端分层、Tauri 桌面壳、React 前端和独立 Tauri 注册机。当前风险不再是“没有分层”，而是旧功能迁移后的页面信息架构、API 边界、运行目录策略和少数大页面/大服务继续增长。

### 2.1 主干结构

- 桌面壳：`apps/export-doc-tauri`，负责 sidecar 启动、运行路径、桌面文件/目录命令、安装交接和窗口生命周期。
- 前端页面：`apps/export-doc-web`，负责主菜单、路由、页面状态、设置中心、报表设计器、单一窗口页面和工具页面。
- 本地 API：`src/ExportDocManager.Api`，负责本机回环 API、鉴权、OpenAPI、后台任务、报表/单一窗口/主数据/设置等端点。
- 应用层：`src/ExportDocManager.Application`，负责用例端口、DTO、分页模型、跨平台业务规则和单一窗口纯规则。
- 基础设施：`src/ExportDocManager.Infrastructure`，负责 EF Core、SQLite/PostgreSQL、运行路径、OCR、报表 PDF、文件系统、授权验签、单一窗口持久化和外部服务适配。
- 领域层：`src/ExportDocManager.Domain`，负责实体、值对象和纯领域规则。
- 授权工具：`apps/license-keygen-tauri`，新的 Tauri/Rust EDM2 离线签名注册机。
- 共享授权编码：`Shared/Security`，保留跨工具共享的许可证编码/验签模型。

### 2.2 代码体量热点

当前最需要警惕的不是“层次不存在”，而是“旧桌面功能迁移后，少数 React 页面、API endpoint 或 Infrastructure 服务承担过多职责”。旧 WinForms 阶段的大量拆分记录仍可作为迁移参考，但当前代码热点应按以下新边界观察：

- `apps/export-doc-web/src/App.tsx`：主菜单已分组，但后续新增一级入口必须先判断是否属于现有分组，避免再次平铺。
- `apps/export-doc-web/src/features/settings/SettingsPage.tsx`：设置中心已分成六类，后续新增设置必须进入对应分类或新分类，不得回到单页堆叠。
- 发票编辑、商品明细、付款/报销、任务中心、Excel 工具、装箱工具和单一窗口页面：保持“高频工作区独立、辅助信息折叠、危险操作确认、长流程进任务中心”的组织方式。
- `src/ExportDocManager.Api/Hosting/*EndpointRouteBuilderExtensions.cs`：端点继续按业务域拆分，避免把认证、设置、报表、单一窗口和工具任务堆进单个启动文件。
- `src/ExportDocManager.Infrastructure/Services/*`：运行路径、授权、报表、单一窗口、OCR、更新和数据库实现保持服务化，新增平台能力必须先抽接口/适配器。
- `apps/license-keygen-tauri`：只作为内部离线签发工具，不应把私钥或注册码生成能力放入主客户端/API sidecar。

## 3. 总体结论

### 3.1 当前架构的优点

- 当前分层整体已经可读，`Domain / Application / Infrastructure / Api / Tauri / Web` 的意图明确。
- 读侧高频页面已经逐步统一到分页查询仓储，`LatestRequestCoordinator` 也把“旧请求不覆盖新请求”收口得比较好。
- 单一窗口已经不再只是零散按钮，而是有实体、跟踪、交接包、回执、协同骨架，方向是对的。
- 测试覆盖面在当前 Tauri/API/Web 迁移阶段已经有了基础护栏，特别是授权、路径策略、API 合同、Web 构建、Tauri check、单一窗口、设置、导出和工具能力都有专项验证。

### 3.2 当前最值得优先处理的 5 个问题

1. 单一窗口保存/跟踪已补显式事务边界，跟踪服务、操作中心骨架与导出预检修复/真实计数回归也已补齐；当前最大回归面开始集中到 Tauri/Web 编辑页布局、字段规则增长和真实客户端交互。
2. `LocalPreferenceService` 已拆到独立 `local-preferences.json`，并补齐多实例同文件写入合并；下一步压力转向大体量编辑器、设置操作模型和发票保存入口守边界。
3. 数据库密钥已不再通过 `DbHelper.CurrentPassword` 静态全局传递；审计拦截器也已完成首轮 `DI + DbContextOptions` 收口，剩余空间主要转向连接配置专职化与大批量审计清理路径。
4. 查询页、发票列表和审计页的读侧路径已统一到共享读仓；Tauri/Web 查询和审计导出走分页读取 + 分段写表 + 固定列宽，剩余压力主要落在 PDF 生成/合并压测、剩余 Excel 导出和其他需持续核查的导出链路。
5. 设置中心、主菜单、任务中心、发票明细工作台和单一窗口页面已经按成熟商用软件方式继续拆分；当前更需要警惕的是 PDF 生成/合并压测、剩余导出链路、发票保存入口、设置操作模型和大页面继续回流堆叠。

## 4.1 启动与依赖装配

涉及文件：

- `src/ExportDocManager.Api/Program.cs`
- `src/ExportDocManager.Api/Hosting/ApiServiceCollectionExtensions.cs`
- `src/ExportDocManager.Api/Hosting/ApiEndpointRouteBuilderExtensions.cs`

- `apps/export-doc-tauri/src-tauri/src/main.rs`

- `apps/export-doc-tauri/src-tauri/src/sidecar.rs`
- `apps/export-doc-tauri/src-tauri/src/desktop_commands.rs`

现状判断：

- API 启动流程、日志、授权、登录、OpenAPI、后台任务和运行目录校验清晰。
- API 组合根已拆成服务注册模块与端点映射模块，避免单个 `Program.cs` 继续膨胀。
- Tauri 主进程已按职责拆成运行路径、sidecar、桌面命令和窗口模块，`main.rs` 只保留生命周期编排。
- 默认数据库配置、程序根、数据根、桌面访问令牌和授权服务都走显式入口，不通过 `null` 或系统用户目录兜底。
- 旧 WinForms 主项目、旧 WinForms 测试项目、旧 WinForms 注册机和旧发布脚本已从解决方案/工作区清理。

高风险点：

- 生命周期审计虽然比单文件阶段清楚很多，但单例服务是否持有可变状态、是否间接依赖 UI 线程，仍需要靠模块边界持续约束。
- 如果后续新模块直接回流到单一入口文件，或把桌面命令、API 端点、Infrastructure 服务和前端页面状态重新混写，装配点仍可能再次膨胀。

可简化点：

- 继续保持“模块扩展 + 薄入口”结构，不再把新注册逻辑塞回 API `Program.cs` 或 Tauri `main.rs`。
- 默认装配入口继续使用显式无参/显式配置入口；测试或专用启动路径传入 `DatabaseConnectionSettings`、`appRoot`、`dataRoot` 时必须传真实对象。
- 新增页面、端点或桌面命令时只追加对应业务模块，不再复制多行注册样板。
- 多接口共享同一个无状态实现时，继续使用 `AddTransientAliases / AddSingletonAliases` 表达“实现 + 接口别名”，避免别名关系分散在多个 lambda 中。

性能/稳定性隐患：

- 主要是维护性风险，不是当前性能瓶颈。
- 但装配点过大，会放大错误配置的排查成本。

建议动作：

- 优先级：中
- 当前 Tauri/API/Web 模块化拆分已落地；下一轮重点是守住模块边界，避免把业务服务、桌面命令、端点注册或测试替身重新塞回单一启动入口。

## 4.2 数据访问与并发控制

涉及文件：

- `DataAccess/AppDbContext.cs`
- `DataAccess/AuditInterceptor.cs`
- `DataAccess/DbHelper.cs`
- `DataAccess/AppDbContextExecution.cs`
- `DataAccess/DatabaseKeyProvider.cs`
- `Models/Entities/Invoice.cs`
- `Models/Entities/Customer.cs`
- `Models/Entities/Exporter.cs`

现状判断：

- `DbContext` 聚合了业务主表、单一窗口实体、审计、工具数据，整体完整。
- 审计和 `RowVersion` 已统一挂在 `AuditInterceptor`，并通过 `DI + DbContextOptions` 显式接入；边界比此前更硬。
- 跨多次 `SaveChanges`、先删后插、批量删除和多表状态回写的事务入口已统一到 `AppDbContextExecution.ExecuteInTransactionAsync(...)`，并通过 EF Core execution strategy 包裹事务，避免 PostgreSQL provider 启用连接重试后与手写事务冲突。
- 当前源码扫描中，业务服务不再直接调用 `BeginTransactionAsync()`；唯一事务开启点在 `AppDbContextExecution`。

高风险点：

- `DbHelper.CurrentPassword` 静态桥接已删除，当前 `SQLite` 密钥改为由 `DatabaseKeyProvider` 实例持有，并在上下文构建时显式传入；这部分错误上下文共享风险已明显下降。
- `AuditInterceptor` 的隐式 `new` 分支已删除；这让行为更透明，但也要求手工/测试上下文必须显式配置拦截器，不能再依赖隐藏兜底。
- `RowVersion` 使用 `[ConcurrencyCheck]`，并发保护是否生效仍取决于调用方是否正确携带原始 `RowVersion`。发票、客户、出口商链路已有回归测试，其余实体仍需按业务重要性继续明确。
- 用户表已补 `DepartmentId / CompanyScope`，发票与付款已补 `OwnerUserId / DepartmentId / CompanyScope`，`BusinessDataAccessPolicy` 已作为统一入口覆盖共享读仓、发票/付款写侧、单据包导入导出、仪表盘、单一窗口操作中心、协同工单和跟踪写入。PostgreSQL 模式下普通账号只能处理本人归属数据，`Admin` 可看全部；共享库首次初始化要求设置 `admin` 初始密码，空密码不能登录共享库；SQLite 单机模式不启用归属过滤并保留本机默认空密码。当前仍不是完整公司级权限后台，因为归属改派、批量补属、公司/部门范围授权、共享库备份/还原和更细管理视图还未补齐。

可简化点：

- 当前已改为上下文构建时显式传入密钥，且审计拦截器已统一通过 `AddInterceptors(sp.GetRequiredService<AuditInterceptor>())` 接入；数据库 provider 与默认 SQLite 文件名已统一到 `DatabaseConnectionSettings`，未知 provider 不再静默落回 SQLite。
- 默认数据库配置入口与自定义入口已继续收紧：`DbHelper.LoadDatabaseSettings()` 承接默认路径，`LoadDatabaseSettings(string)` 要求真实路径，`ConfigureDbContextOptions(...)` 和 `DatabaseModeHelper` 不再接受空配置对象。
- 明确哪些实体必须启用乐观并发，哪些只做普通更新，不要“看起来都支持，实际上只在部分页面真正生效”。
- 公司级 PostgreSQL 共享生产前，应在现有用户管理入口、归属字段和统一过滤基础上，补归属改派/批量补属、部门/公司范围授权、共享库备份/还原和对应测试。

性能/稳定性隐患：

- `AuditLogService` 历史清理已改成数据库侧分批删除；审计导出列宽已固定，剩余风险主要在其他大批量导出链路的写表策略和前台耗时反馈。
- 审计序列化异常虽然做了兜底，但发生时会损失字段级可追溯性。

建议动作：

- 优先级：中
- 当前 DI 收口、发票/客户/出口商并发回归、审计批量清理、事务执行策略统一、数据库配置入口收紧、发票/付款归属字段、PostgreSQL 普通账号统一过滤、管理员用户管理入口和误建用户删除保护已落地；SQLite/PostgreSQL provider 切换、归属过滤、单一窗口派生视图过滤、管理权限和用户删除边界已有专项测试。下一轮继续围绕更多业务实体并发边界、归属改派/批量补属、公司/部门范围授权和 PostgreSQL 备份/还原推进。

- `LocalPreferenceService` 已按偏好文件路径做进程内串行写入，并在保存前重新合并磁盘状态；偏好键加载和保存时按修剪后大小写不敏感口径处理。
- `AtomicFileHelper.ReplaceFile()` 已对 Windows 短暂文件占用增加共享重试，设置/偏好/导出调用层不需要各自复制 IO 重试样板。

高风险点：

- 业务设置与本机偏好已不再争抢同一个文件，原先“最后写入覆盖前者修改”的核心竞态已基本消除。
- 多个服务实例同时写同一个偏好文件时，不同键互相覆盖的风险已通过保存前合并和专项测试压低。
- 原子替换仍可能遇到长期外部占用，但短暂占用已由共享 helper 重试；最后一次失败会保留真实异常便于定位。
- 残余风险主要变成两类：本机偏好文件的增长管理，以及是否需要为未来偏好结构演进补更明确的版本化策略。
- 布局偏好链路已不再有空服务 no-op 分支；后续新增窗体如果需要列宽或分栏记忆，必须显式注入本机偏好服务。

可简化点：

- `LocalPreferenceService` 当前已足够轻量，后续保持“单文件 + 合并写盘 + 原子替换 + 小状态对象”即可，不建议再回流到 `SettingsService`。
- `DataGridViewLayoutPreferenceBinder / SplitContainerPreferenceBinder` 当前只保留有效偏好链路，不再承接“缺服务就禁用”的运行时胶水。
- 如果未来本机偏好继续增多，可再按窗口/模块分桶或补版本字段，但不必现在过度设计。

性能/稳定性隐患：

- 当前主要隐患已从“共享设置互相覆盖/多实例写偏好丢键”转为“偏好状态数量持续增长时的可观测性与演进管理”，紧迫度明显下降。

建议动作：

- 优先级：中
- `appsettings.json` 与 `local-preferences.json` 拆分已落地。
- 保持原子落盘统一走 `AtomicFileHelper`，不要在业务服务里复制临时文件替换与重试逻辑。
- 下一步可把注意力转到 `CustomsCooDocumentEditorViewModel / Form` 大节点，以及发票保存入口守边界。

- 主窗口和设置页后台预热已接入幂等关闭取消令牌，关闭/释放时不会再次访问已释放的 `CancellationTokenSource`；共享页签 helper 会自动优化后续新增的 `TabPage` 与页内懒加载子控件，并且对同一 `TabPage` 去重，避免显式优化和统一样式入口重复递归。初始加载入口由主工作区统一调用，报表设计器 WebView2 初始化也已后移到窗口显示后的消息循环，避免构造阶段一次性堆叠。

## 6. 最终判断

这个项目当前最核心的事实是：

- 架构主方向是对的，很多通用骨架已经比早期稳很多。
- 真正需要警惕的，不再是“有没有分层”，而是“少数超大模块已经开始承担过多职责”。
- 后续如果继续加功能而不守住显式依赖、分页导出和模块拆分边界，维护成本会重新上升。

因此，最推荐的路线不是“大重构”，而是：

1. 守住事务、并发、显式依赖和分页读侧边界。
2. 继续拆单一窗口编辑器、设置操作模型和其他仍偏大的窗体/ViewModel。
3. 持续压测导出/OCR/报表这类性能型链路。

这条路线最符合当前项目“保持代码简洁高效、性能稳定、尽量少回归”的目标。
