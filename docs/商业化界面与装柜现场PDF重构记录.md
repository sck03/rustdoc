# 商业化界面与装柜现场 PDF 重构记录

> 日期：2026-07-16  
> 范围：普通用户文案、系统维护信息架构、普通/窄窗口布局、装柜模拟现场 PDF

## 1. 本轮目标与结论

本轮不改变数据库模型、业务数据归属、装柜分析算法或三产品版本权限，只收口普通用户界面和装柜结果交付能力。

- 普通业务页面不再直接展示 `storagePolicy`、程序根、`App_Data`、服务器路径限制等开发/审查说明。此类内容只保留在管理员“运行检查”任务中。
- 维护工具从连续堆叠的大页面调整为“团队库 / 数据归属 / 运行检查 / 问题诊断”四个任务页签，每次只呈现当前任务。
- 英文确认口令不再要求普通管理员手工输入。界面改为中文影响说明和确认弹窗，服务端既有确认字段继续由前端按确认结果提交，安全门禁没有降低。
- 维护表单改为自动换列网格，`861–1180px` 普通窗口和更小窗口都不再依赖固定五列最小宽度；原数据库所有者移入折叠高级选项，数据归属复选项允许换行。
- 装柜模拟增加 PDF 导出。PDF 定位为“现场装柜作业单”，不复制整张操作页面，不包含可旋转 Three.js 3D、按钮、状态栏或技术说明。

## 2. 装柜现场 PDF 内容边界

PDF 固定包含：

1. 方案名称、柜型和生成时间。
2. 装载件数、未装件数、估算柜数、体积/重量利用率和重心状态。
3. 现场提示：核对柜型、柜头到柜门方向、颜色对应关系、未装件和重心风险。
4. 伪 3D 装柜效果图。
5. 俯视图、侧视图、柜门图和颜色图例。
6. 货物名称、尺寸、单重、数量、区域和托盘信息清单。

明确排除：

- 可旋转 Three.js 交互 3D。它用于电脑端检查视角，打印后没有交互价值。
- 方案保存、柜型维护、自动刷新、分析、显示模式等操作控件。
- API 状态、运行目录、缓存目录、存储策略和 desktop token 等技术信息。

## 3. 代码与解耦设计

| 职责 | 实现 |
| --- | --- |
| 装柜业务状态和分析 | 继续由 `ContainerPackingPanel.tsx` 与既有 API 负责 |
| 页面编排与导出入口 | `ContainerPackingWorkspace.tsx` |
| PDF 捕获、分页、命名和平台保存 | 独立 `containerPackingPdfExport.ts`，点击时动态加载 |
| 可视化 | 继续复用既有伪 3D、正投影和 Three.js 组件；PDF 仅选择前两类静态视图 |
| 桌面保存 | Tauri `select_save_pdf_path` 选择用户路径，`save_pdf_file` 校验 PDF 头、扩展名、大小和父目录后写入 |
| 浏览器保存 | 内存 Blob + 标准下载，不向服务器提交绝对路径，不保留服务器副本 |

前端新增 `html2canvas` 与 `jspdf`，均为免费开源库，并被拆为按需异步 chunk；不进入首屏主包。依赖安装在工作区 `apps/export-doc-web/node_modules`，安装缓存显式使用工作区 `.codex-runtime/npm-cache`。客户运行时只使用已打包静态资源，不依赖系统 C 盘 npm 目录或全局 Node.js。

## 4. 路径、数据库与多平台边界

- 本轮没有新增数据库表、迁移、JSON 业务副本、LocalStorage、SessionStorage 或 IndexedDB。
- 装柜项目和柜型继续只写当前运行数据库；PDF 读取当前 React 内存中的分析结果和已渲染视图。
- 浏览器版 PDF 直接下载到浏览器管理的用户下载位置，服务器不创建 `Exports/Browser` 副本。
- Tauri 桌面版只写用户在原生保存对话框明确选择的 `.pdf`；不使用 `Path.GetTempPath`、AppData、ProgramData 或默认 C 盘目录。
- Rust 保存命令拒绝非 `.pdf`、非 `%PDF-` 文件、超过 25 MB 的内容和不存在的父目录，避免把通用任意文件写入能力暴露给页面。

## 6. 2026-07-16 截图反馈复核

### 6.1 维护工具固定为四个任务入口

“数据归属”此前仍被账号管理权限条件隐藏，导致单证员版只看到三个页签。现在四个页签在所有桌面版本中固定显示，避免同一套产品的信息架构随权限变化而缺项：

- 全功能团队管理员：显示真实的数据归属改派表单。
- 单机单用户或无账号管理权限版本：显示“当前无需改派”的说明，不提供来源人员、接手人员和提交操作。

该处理只统一导航，不扩大权限，也不增加为旧数据兼容的分支。

### 6.2 邮件未配置状态改为普通用户可理解的文案

原来的 `-:587 / -` 是“空服务器 + 默认端口 587 + 空发件人”的机械拼接，不是有效 SMTP 地址。现在未配置时直接显示：

- `SMTP 未配置`
- `请先配置邮件服务器和发件人信息`

页面同时提供进入邮件设置的按钮；只有配置完整后才显示 `服务器:端口 · 发件人` 摘要。

### 6.3 PDF 失败原因与可靠性修复

截图中的失败不是文件太大。最新真实自动化方案生成的 PDF 为 `102,288 bytes`（界面约 `100 KB`），远低于 `10 MB` smoke 门禁和桌面 `25 MB` 写入上限。问题更符合旧桌面壳尚未包含新 `save_pdf_file` 命令：PDF Blob 已经生成，但最后一步原生写文件失败，被旧逻辑笼统显示为“生成失败”。

当前保存策略为：

1. 先在内存生成一次现场作业 PDF。
2. 桌面原生命令可用时，保存到用户选择的位置。
3. 原生命令缺失或写入失败时，复用同一 Blob 自动转为普通下载，并显示文件大小。
4. 只有画布捕获或 PDF 组装失败，才显示具体生成错误。

同时将导出宽度调整为 `960px`、捕获倍率设为 `1.15`、分页图片质量设为 JPEG `0.78`。PDF 内容边界保持不变：保留现场真正有用的伪 3D、三向投影、装载摘要、风险提示、货物清单和颜色图例；可旋转 3D 及页面操作控件仍不进入 PDF。

### 6.4 验证结论

- 四个维护页签固定存在，无团队权限版本的“数据归属”不会触发改派 API。
- 邮件未配置 smoke 明确断言不再出现 `-:587 / -`。
- 装柜 smoke 模拟 `save_pdf_file` 不存在，仍得到 `application/pdf`、`.pdf` 文件名、`102,288 bytes` 非空内容和“已改为普通下载”成功提示。
- Web 生产构建、设置模型测试、Tauri 桌面命令测试和 Full/Document Debug 编译均通过。

本次反馈修复完成度：`100%`。

## 7. A4 成品、另存为与低扰动编辑复核

用户提供的实际 PDF 证明旧版并非“文件太大”，而是导出画布本身已经裁掉右侧内容，同时因为高度只超出一页约数个像素而生成无意义尾页。本轮改为：

1. 在页面外创建独立作业单副本，不受当前窗口宽度和滚动容器约束。
2. 以 `960px` 作为清晰度基准，同时读取真实内容宽度，完整捕获后再等比缩放到 A4 可打印区。
3. 摘要、伪 3D、俯视/侧视/柜门图和货物表格都允许收缩、换行，不再把右侧内容推出画布。
4. 接近一页时整体轻微缩小到一张 A4；长清单才分页，并均衡每页内容，避免几像素尾页。

桌面保存失败的真实原因是 Tauri 权限白名单漏掉 `save_pdf_file`，不是 editions 脚本没有重新编译。权限现已补齐，并增加注册命令/权限清单一致性门禁。桌面端现在只采用原生“另存为”：用户选哪里就保存到哪里；失败时明确提示重新选择，不再自动落入默认下载文件夹。

权限门禁已审计 Rust 主程序当前注册的全部 `28` 个桌面命令，白名单同为 `28` 个，没有发现第二个漏授权命令。检查脚本在三版本构建、单版本构建和版本验证开始时运行，避免完成长时间编译后才发现权限问题。

三版本构建检测到旧运行目录的 `App_Data`/`logs` 时不再直接报错关闭窗口，而是列出路径并询问：直接回车或 `Y` 继续清理覆盖，`N` 停止。底层清理仍要求显式确认参数，防止其他脚本绕过提示。

装柜编辑交互同步调整为常见商业软件方式：

- 自动分析默认关闭，避免用户刚进入页面或连续录入时频繁计算。
- 用户开启后，停止输入约 `0.9` 秒才运行一次，连续按键只保留最后一次任务。
- 编辑期间继续显示上一版装柜结果，不清空伪 3D 和三向视图，不造成页面高度跳变。
- 状态栏用“内容已修改 / 正在更新 / 已完成”反馈；需要立即查看时直接点击“立即刷新”。

最新真实装柜 smoke 结果为 `application/pdf`、`110,999 bytes`、`1 页`；PDF 保存模拟未触发普通下载，编辑 250ms 后旧结果仍存在，延迟后自动分析只完成一次。该增量完成度为 `100%`。
